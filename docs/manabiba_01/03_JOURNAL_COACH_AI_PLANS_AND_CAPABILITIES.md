# マネジメント日誌 — 人コーチ・AI・サブスクコース（能力マトリックス）

## 📋 目的

マネジメント日誌（`journal_daily` / `journal_weekly` / 将来 `journal_monthly`）について、**人間パーソナルコーチ**と **AI コーチ**の役割を区別し、**サブスクのコース（プラン）**で ON/OFF を切り替えるための**正本**とする。

- **関連**: [03_A11_COACH_SHARING_SCHEMA_DRAFT.md](./03_A11_COACH_SHARING_SCHEMA_DRAFT.md)（アファメーション共有）、[04_TRIAL_28_IMPLEMENTATION_DECISIONS.md](./04_TRIAL_28_IMPLEMENTATION_DECISIONS.md) §5、[01_ROLES_AND_SUBSCRIPTION_DESIGN.md](./01_ROLES_AND_SUBSCRIPTION_DESIGN.md)
- **実装（Vertex・トライアル朝晩の「Aiコーチからのコメント」PoC）**: [../VERTEX_AI_TRIAL_IMPROVEMENT.md](../VERTEX_AI_TRIAL_IMPROVEMENT.md)（環境変数・IAM・モデル ID・REST エンドポイント・短文時の再生成・トークン付記）

## 1. 確定した前提

| 項目 | 内容 |
|------|------|
| **人コーチへの共有データ** | **週次・月次ドキュメントに限定**。`journal_daily` は人コーチに**送らない**・**Firestore ルールで read 不可**（クライアント本人のみ）。 |
| **AI** | 日次・週次・月次それぞれで質問・回答・（文脈としての）共有を**プロダクト上は許容**。実装詳細は **AI 実装フェーズ**で決定。 |
| **AI への情報提供** | **クライアントが UI 上で明示的に操作（例: ボタン）したとき**に、当該期間の情報を API に載せる。**利用者の選択と責任**を前提とする（同意文面・表示は AI 実装時に詳細化）。 |
| **動的ルール** | Firestore ルール内で「担当人数」などを**都度計算しない**。**コース ID または能力フラグ**を `users/{uid}`（等）に持ち、**運用でコースを切り替える**。 |

**補足（サブスク 2 コースとの対応表）**: 検討メモで用いた「AI のみ／AI＋パーソナル」の**共有単位・質問・コメント**の整理は、本書 **§1（前提）** と **§2（マトリックス）** の組み合わせで表現する（数式の二重定義にしない）。

## 2. 能力マトリックス（現行仕様）

「共有」＝そのチャネルで**対象期間のデータを相手に見せる／AI リクエストに含めてよい**こと（人コーチは Firestore read、AI は API 経由）。

| 区分 | 人コーチ・質問 | 人コーチ・回答 | 人コーチ・共有 | AI・質問 | AI・回答 | AI・共有 |
|------|----------------|----------------|----------------|----------|----------|----------|
| **日次** | なし | なし | なし | あり | あり | あり |
| **週次** | なし | なし | あり | あり | あり | あり |
| **月次** | あり | あり | あり | あり | あり | あり |

- **週次（人コーチ）**: 共有のみ（閲覧）。**週単位の人コーチとの質問・回答スレッドは持たない**（負荷・運用で月次に集約）。
- **月次（人コーチ）**: 共有に加え、**質問・回答あり**。枠の単位は **暦月 1 回**（`monthKey` は **東京（Asia/Tokyo）基準の `YYYY-MM`** とする想定）。**未送信分の翌月への繰越はなし**（当月未使用分は失効）。
  - **「月1回」の定義（確定）**: **クライアントがパーソナルコーチへ「質問」を送信できる回数が、暦月 1 回まで**（質問送信が消費トリガー）。共有 ON だけでは消費しない。

### 2.1 週次共有と「月のみ対話」（検討メモの案 A / 案 B）

検討時の言い方との対応は次のとおり。**案 A と案 B はどちらも §1 単体では区別できず**、差は **§2 の週次「人コーチ・共有」の有無**で表す。

| 検討メモ | 意味 | 本書での表現 |
|----------|------|----------------|
| **案 A**（週は閲覧用・月で対話） | 週次も共有 ON ならパーソナルコーチは**読める**。質問・回答の**正式枠は月報の暦月 1 回**のみ。 | **§2 現行どおり**（週次: 共有＝あり、質問・回答＝なし）＋ §1 の週次も人コーチ対象。**これを採用する。** |
| **案 B**（パーソナルには週は送らない） | 人コーチに届くのは**月次のみ**。週次は本人／AI のみ。 | §2 の週次行で **人コーチ・共有＝なし** に変更したパターン。別 `courseId` または `coachCapabilities` の上書きで表現する。 |

**結論**: 「週次までデータは見せるが週ではやり取りしない」という整理は、**§1（週・月だけ人コーチに共有しうる）＋ §2（週は共有のみ）**の組み合わせで**案 A と同義**に書ける。案 B に切り替える場合はマトリックスの週次「共有」だけを false にすればよい（§2.2 の拡張性の考え方どおり）。

### 2.2 将来拡張

マトリックスの各セルを **true/false** で持てるようにしておけば、例えば「週次まで人コーチが対応する上位コース」「日次も人コーチが見るプラン」などに**スキーマ変更なし**で拡張できる。Firestore ルールは **`users/{uid}` の能力フラグ**または**コース ID → 能力の対応表（クライアント／Functions で解決）**を参照する形が現実的。

## 3. サブスク・コース定義（案）

コースは **識別子 `courseId`（文字列）** で管理する。決済連携時は外部プロダクト ID とマッピングする。

### 3.1 コース一覧（初期案）

| `courseId` | 表示名（仮） | 人コーチ割当 | 説明 |
|------------|--------------|--------------|------|
| `ai_only` | AI のみ | なし | AI 機能のみ。人コーチ用の `journal_weekly` / `journal_monthly` read は**ルール上許可しない**（能力はすべて false 扱い）。 |
| `ai_plus_personal` | AI ＋ パーソナルコーチ | あり（`coach_client_assignments` で active） | 下記 §4 の能力を適用。月報に対するパーソナル向けやり取りは **§2 の暦月 1 回・繰越なし**に従う。アファメーションの人コーチ共有（A-11）は既存どおり**別枠**（クォータ・回数は A-11 の `coachShareQuotaPerMonth` 等で管理。本書の月報 1 回とは独立して数える）。 |

### 3.2 追加コース例（将来・名前のみ）

| `courseId`（例） | 用途のイメージ |
|------------------|----------------|
| `ai_plus_personal_weekly_qa` | 週次まで人コーチの質問・回答を有効にする（マトリックスの週次「質問・回答」を true に変更するだけで表現可能） |
| `enterprise_daily_review` | 日次の人コーチ共有・QA を有効にする拡張用 |

実装では **`courseId` を増やす**か、管理画面で **能力オブジェクトを上書き**するかを選べるようにすると運用が柔軟。

## 4. 能力オブジェクト（DB・ルールのたたき台）

`users/{uid}` に持つか、サブドキュメント `users/{uid}/subscription/summary` 等に持つ想定。**ルールが読める場所**に置く。

### 4.1 構造（論理モデル）

```text
coachCapabilities.human.journal.daily   = { share, question, reply }  // いずれも boolean
coachCapabilities.human.journal.weekly  = { share, question, reply }
coachCapabilities.human.journal.monthly = { share, question, reply }
coachCapabilities.ai.journal.daily      = { share, question, reply }
coachCapabilities.ai.journal.weekly     = { share, question, reply }
coachCapabilities.ai.journal.monthly    = { share, question, reply }
```

### 4.2 コース別の初期値（§2 マトリックスの写し）

**`ai_only`**

- `human.journal.*` … すべて `{ share: false, question: false, reply: false }`
- `ai.journal.*` … すべて `{ share: true, question: true, reply: true }`

**`ai_plus_personal`**

| period | human.share | human.question | human.reply |
|--------|-------------|----------------|-------------|
| daily | false | false | false |
| weekly | true | false | false |
| monthly | true | true | true |

- `ai.journal.*` … すべて `{ share: true, question: true, reply: true }`

**実装メモ**: 正本を **`courseId` のみ**保存し、クライアント・ルール・Functions が **コード上のマップ**で能力を展開してもよい（DB に冗長なオブジェクトを書かない）。逆に、**管理画面で個別上書き**したい場合は `coachCapabilities` を実体として保存。

## 5. Firestore ルールとの関係

| やりたいこと | 方針 |
|--------------|------|
| 人コーチが `journal_daily` を read しない | **パス単位で禁止**（現状どおり本人のみ）。コースに依存しない。 |
| 人コーチが `journal_weekly` を read する | **共有フラグ ON** かつ **active 割当** かつ **`human.journal.weekly.share == true` 相当**（`courseId` 解決でも可）。 |
| 人コーチが週次にコメントを書く | マトリックス上 **reply なし**のため、**`coach_comment_versions` 等の write 用パスを週次には置かない**／ルールで **create 禁止**が簡潔。 |
| 人コーチが月次にコメントを書く | **月次ドキュメント配下**に A-11 と同型の `coach_share_rounds` / `coach_comment_versions` を用意し、`human.journal.monthly.reply` が true のコースだけ write 可。 |

動的な「クライアント数が N 以上なら週次オフ」は **ルールではやらず**、**コース変更**または **バッチで `courseId` / フラグを更新**する運用とする（ユーザー回答どおり）。

## 7. データ構造（週次・月次）と A-11 の関係

- アファメーション: 既存どおり `users/{uid}/affirmations/{id}/coach_share_rounds/...`
- **週次**: `users/{uid}/journal_weekly/{weekStartKey}` に共有メタ（例: `sharedWithCoach`）＋必要なら **同型のラウンド／コメント版**（人コーチ reply が false ならコメント用サブコレクションは未使用またはルール閉鎖）
- **月次**: `users/{uid}/journal_monthly/{monthKey}` を同様に定義（実装タイミングで [03_FIRESTORE_DATABASE_STRUCTURE.md](./03_FIRESTORE_DATABASE_STRUCTURE.md) に追記）

詳細パス・フィールド名は週次・月次 UI 実装とあわせ [03_A11_COACH_SHARING_SCHEMA_DRAFT.md](./03_A11_COACH_SHARING_SCHEMA_DRAFT.md) に**別節で追記**する想定。

### 7.1 クォータの「暦月」とローリング、A-11 との関係

| 疑問 | 回答 |
|------|------|
| 別カウンタ例（`personalCoachMonthKey` / 使用回数）は**ローリング 30 日**の話か | **いいえ。** 本書 §2 で確定したとおり、パーソナル月報の枠は **暦月（東京 `YYYY-MM`）1 回・繰越なし**。カウンタは **A-11 の `coachShareMonthKey` / `coachShareUsedThisMonth` と同型の考え方**（月キーが変わったら使用回数をリセット）でよい。 |
| データスキーマはアファメーションと同様か | **同じ思想**でよい。親ドキュメント（週／月）に共有メタ・未読・最終送信時刻等、やり取りは **`coach_share_rounds` / `coach_comment_versions`** を親の下にぶら下げる A-11 型。週次は現行仕様では人コーチの **reply なし**のため、コメント用サブコレクションはルールで閉じるか省略。 |
| アファメーションの `coachShareQuotaPerMonth` とどう分けるか | **別フィールド・別カウンタ**とする（例: `journalMonthlyPersonalCoachMonthKey` / `journalMonthlyPersonalCoachUsedThisMonth` など、名称は実装時に確定）。アファメ送信 1 回と月報の人コーチ 1 枠は**別物**なので、共有してはいけない。 |
| 「月1回」は何を数えるか | **クライアントがパーソナルコーチへ「質問」を送信した回数**（暦月 1 回まで）。 |
| AI の上限・条件 | **別枠**（§6）。API レート・1 日 N 回等は **AI 実装時**に決定。本 §7 の暦月カウンタは **パーソナルコーチ × 月報**用。 |

**今までの流れとの矛盾**: なし。暦月 1 回・繰越なし（§2）と、暦月キーでリセットするカウンタ（§7.1）は一致している。AI は別経路（§1・§6）。
