# 外部仕様書作成ガイド

紙ベースのツールをWeb化するための外部仕様書作成手順です。

## 📋 作成の流れ

### ステップ1: 現状分析（紙ベースの詳細把握）

#### 1.1 現状のツールを整理
以下の3種類について、それぞれ詳細を記録します：

1. **テキスト（紙レベル）**
   - 実際のテキストを確認
   - 内容、構成、ページ数を記録
   - 使用タイミングを確認

2. **ワークシート（フォームへの記入）**
   - 実際のワークシートを確認
   - 記入項目をすべてリストアップ
   - 記入フローを確認
   - 所要時間を測定

3. **毎日のチェックリスト**
   - 実際のチェックリストを確認
   - チェック項目をすべてリストアップ
   - チェックフローを確認
   - 所要時間を測定

#### 1.2 現状のプロセスを可視化
- プロセスフロー図を作成
- 各ツールの関係性を明確化
- 課題を洗い出し

#### 1.3 課題の整理
- 紙ベースの課題をリストアップ
- 影響度と優先度を評価
- Web化で解決できる課題を特定

### ステップ2: 要件定義（Web化の要件を明確化）

#### 2.1 Web化の目的を明確化
- なぜWeb化するのか
- 何を解決したいのか
- 期待する効果は何か

#### 2.2 機能要件の定義
- 必須機能（Must Have）
- 推奨機能（Should Have）
- 将来機能（Could Have）

#### 2.3 非機能要件の定義
- パフォーマンス要件
- セキュリティ要件
- ユーザビリティ要件

### ステップ3: 比較分析（現状 vs Web版）

#### 3.1 機能比較表の作成
- テキスト機能の比較
- ワークシート機能の比較
- チェックリスト機能の比較

#### 3.2 改善点の明確化
- 各機能の改善点を記録
- 新規追加機能を記録
- 削除・変更する機能を記録

### ステップ4: 詳細仕様の作成

#### 4.1 機能仕様の詳細化
- 各機能の詳細を記述
- 画面設計
- データ項目定義
- 処理フロー

#### 4.2 ユーザーストーリーの作成
- ユーザーの視点から機能を記述
- 受け入れ基準を定義

#### 4.3 ユースケースの作成
- ユースケース図を作成
- 主要ユースケースの詳細を記述

### ステップ5: レビュー・承認

#### 5.1 内部レビュー
- 開発チームでのレビュー
- デザイナーでのレビュー
- QAでのレビュー

#### 5.2 ステークホルダーレビュー
- クライアントとの確認
- 実際のユーザーでの確認
- 承認取得

## 🎯 作成時のポイント

### 現状分析のポイント

1. **実際のツールを手元に置く**
   - 紙ベースのツールを実際に見ながら記述
   - 写真を撮って記録

2. **使用シーンを確認**
   - 実際のセッションでどのように使われているか
   - クライアントの反応を記録

3. **課題を具体的に記述**
   - 「使いにくい」ではなく「手書きで読みにくい」など具体的に
   - 影響度を数値化できるものは数値化

### 比較分析のポイント

1. **1対1で対応付け**
   - 紙ベースの各機能とWeb版の機能を1対1で対応付け
   - 対応できない機能は明確に記述

2. **改善点を明確に**
   - なぜ改善するのか理由を記述
   - 改善による効果を記述

3. **新規機能の追加理由**
   - なぜ新規機能が必要なのか
   - 紙ベースでは実現できなかった理由

### 詳細仕様のポイント

1. **具体性を重視**
   - 抽象的な表現を避ける
   - 具体例を多く含める

2. **画面設計は視覚的に**
   - ワイヤーフレームやモックアップを作成
   - 画面遷移図を作成

3. **データ項目は漏れなく**
   - 紙ベースのすべての項目を網羅
   - 新規追加項目も明確に

## 📝 テンプレートの使い方

### 1. テンプレートをコピー
```bash
cp docs/EXTERNAL_SPECIFICATION_TEMPLATE.md docs/[機能名]_外部仕様書.md
```

### 2. 各セクションを埋める
- 必須セクションから順に埋める
- 不明な点は「要確認」と記入して後で確認

### 3. 図表を作成
- PlantUMLで図を作成
- 必要に応じてワイヤーフレームを作成

### 4. レビュー・修正
- チームでレビュー
- フィードバックを反映
- 繰り返し改善

## 🔄 継続的な改善

### 定期的な見直し
- 月1回、仕様書を見直し
- 実装との整合性を確認
- 変更があれば更新

### バージョン管理
- Gitで管理
- 変更履歴を記録
- 承認プロセスを明確化
