# プロトタイプ機能追加ガイドライン

## 📋 目的

このドキュメントは、インタラクティブプロトタイプ（HTML/CSS/JS）の段階で、どの程度までの機能追加・修正が適切かを定義します。

---

## 1. プロトタイプの目的と範囲

### 1.1 プロトタイプの主な目的

- **UI/UXの確認**: レイアウト、デザイン、操作性の確認
- **画面遷移の確認**: ナビゲーション、画面間の流れの確認
- **インタラクションの確認**: ボタンクリック、フォーム入力、モーダル表示等の動作確認
- **ステークホルダーとの合意形成**: デザインと機能の方向性を共有

### 1.2 プロトタイプで**確認しない**こと

- サーバー側の処理（認証、データ保存、API連携等）
- データの永続化（Firestore等への保存）
- 本番環境でのパフォーマンス
- セキュリティ要件の実装

---

## 2. プロトタイプ段階で追加・修正できる機能

### 2.1 ✅ 推奨：追加すべき機能

#### 2.1.1 UI/UX確認に必要な機能

| 機能 | 説明 | 実装方法 | 例 |
|------|------|----------|-----|
| **フォームバリデーション** | クライアントサイドでの入力チェック | JavaScript（正規表現、必須チェック等） | メール形式チェック、文字数制限 |
| **モーダル表示/非表示** | ダイアログ、確認画面の表示 | JavaScript（`display: none/block`の切り替え） | 削除確認モーダル、詳細表示モーダル |
| **タブ切り替え** | タブナビゲーション | JavaScript（既に実装済み） | マイページのタブ |
| **アコーディオン** | 折りたたみ表示 | JavaScript（`classList.toggle`） | FAQ、詳細説明の展開/折りたたみ |
| **スムーススクロール** | アンカーリンクのスムーズな移動 | CSS（`scroll-behavior: smooth`、既に実装済み） | ホームページのセクション移動 |
| **条件分岐による表示切り替え** | 状態に応じたUIの変化 | JavaScript（`if/else`、`classList`） | ログイン状態による表示切り替え（モック） |
| **アニメーション・トランジション** | ホバー効果、フェードイン等 | CSS（`transition`、`animation`） | ボタンホバー、カードの表示アニメーション |
| **レスポンシブデザイン** | 画面サイズに応じたレイアウト変更 | CSS（Media Queries、Flexbox、Grid） | モバイル/タブレット/デスクトップ対応 |

#### 2.1.2 モックデータを使った機能

| 機能 | 説明 | 実装方法 | 例 |
|------|------|----------|-----|
| **一覧表示** | モックデータの配列を表示 | JavaScript（`Array.map()`、`innerHTML`） | PDCAエントリ一覧、ニュース一覧 |
| **検索・フィルタリング** | クライアントサイドでの検索 | JavaScript（`Array.filter()`） | テーマ選択の検索、PDCAの絞り込み |
| **ソート** | クライアントサイドでの並び替え | JavaScript（`Array.sort()`） | 日付順、優先度順のソート |
| **ページネーション** | モックデータのページ分割 | JavaScript（配列の分割、表示制御） | 一覧のページ送り |

#### 2.1.3 ローカルストレージを使った簡易的な状態管理

| 機能 | 説明 | 実装方法 | 例 |
|------|------|----------|-----|
| **入力内容の一時保存** | フォーム入力の一時保存 | `localStorage.setItem/getItem` | 目標設定フォームの下書き保存（モック） |
| **表示状態の保持** | タブの選択状態等の保持 | `localStorage` | 最後に選択したタブの記憶 |

## 2.2 レスポンシブデザインの扱い

### 2.2.1 モックアップとプロトタイプの役割分担

レスポンシブデザインは、**モックアップとプロトタイプの両方で対応**しますが、それぞれ異なる役割があります。

#### モックアップ（Figma等）での対応

| 役割 | 説明 | 実施内容 |
|------|------|----------|
| **デザインの確認** | 主要ブレークポイントでのビジュアルデザインを確認 | Mobile（< 768px）、Tablet（768-1024px）、Desktop（> 1024px）の3つのサイズでデザインを作成 |
| **レイアウトの検討** | 画面サイズに応じたレイアウト変更を検討 | カラム数の変更、ナビゲーションの表示方法（ハンバーガーメニュー等）を検討 |
| **ビジュアルデザインの確定** | 各ブレークポイントでのデザインを確定 | フォントサイズ、余白、コンポーネントサイズを確定 |

**実施タイミング**: デザイン確定時（モックアップ作成時）

#### プロトタイプ（HTML/CSS/JS）での対応

| 役割 | 説明 | 実施内容 |
|------|------|----------|
| **実際の動作確認** | ブラウザで実際に画面サイズを変更して動作を確認 | CSS Media Queriesを実装し、実際のレスポンシブ動作を確認 |
| **インタラクションの確認** | 各画面サイズでの操作性を確認 | ハンバーガーメニューの開閉、タッチ操作の確認等 |
| **実装方針の検証** | 実装段階での実装方針を検証 | Flexbox、Grid、Media Queriesの実装方法を検証 |

**実施タイミング**: インタラクション確認時（プロトタイプ作成時）

### 2.2.2 推奨されるワークフロー

```
1. モックアップ作成
   ↓
   Mobile/Tablet/Desktop の3つのサイズでデザインを作成
   ↓
2. デザインレビュー
   ↓
   各画面サイズでのデザインを確認・承認
   ↓
3. プロトタイプ実装
   ↓
   CSS Media Queriesでレスポンシブを実装
   ↓
4. 動作確認
   ↓
   ブラウザで実際に画面サイズを変更して動作を確認
```

### 2.2.4 確認方法

プロトタイプでのレスポンシブ確認は、以下の方法で実施します：

1. **ブラウザの開発者ツール**
   - Chrome DevTools: `F12` → デバイスツールバー（`Ctrl+Shift+M`）
   - 主要デバイスの画面サイズで確認

2. **実際のデバイス**
   - スマートフォン、タブレットで実際にアクセスして確認
   - タッチ操作の確認

3. **主要ブレークポイント**
   - Mobile: `< 768px`
   - Tablet: `768px - 1024px`
   - Desktop: `> 1024px`

### 2.2.5 まとめ

| 項目 | モックアップ | プロトタイプ |
|------|------------|------------|
| **目的** | デザインの確認・確定 | 実際の動作確認 |
| **実施内容** | 3つの画面サイズでデザイン作成 | CSS Media Queriesで実装 |
| **実施タイミング** | デザイン確定時 | インタラクション確認時 |
| **確認方法** | デザインレビュー | ブラウザでの動作確認 |

**結論**: 
- **モックアップ**: デザインの確認・確定（3つの画面サイズでデザイン作成）
- **プロトタイプ**: 実際の動作確認（CSS Media Queriesで実装）

両方が必要ですが、**まずモックアップでデザインを確定し、その後プロトタイプで実装・確認**する流れが推奨されます。

## 2.3 多言語化（i18n）の扱い

### 2.3.1 プロトタイプ段階での多言語対応

プロトタイプ段階では、**和文版と英文版の両方を作成**し、レイアウトやフォントサイズの調整を確認します。

#### 推奨されるワークフロー

```
1. 和文版プロトタイプの作成
   ↓
   すべての画面を和文で作成・確認
   ↓
2. 英文版プロトタイプの作成
   ↓
   和文を英文に一括翻訳
   ↓
3. レイアウト・フォントサイズの調整
   ↓
   英文の長さに応じてマージン、フォントサイズ、レイアウトを調整
   ↓
4. 両方のバージョンを確認
   ↓
   和文版と英文版の両方でUI/UXを確認
```

### 2.3.2 英文版作成時の注意点

#### テキストの長さの違い

| 要素 | 和文 | 英文 | 調整が必要な項目 |
|------|------|------|----------------|
| **見出し** | 短い | 長くなる傾向 | フォントサイズ、行の高さ |
| **本文** | コンパクト | 1.2〜1.5倍の長さ | マージン、パディング、行間 |
| **ボタン** | 短い | 長くなる傾向 | ボタン幅、パディング |
| **ナビゲーション** | 短い | 長くなる傾向 | ナビゲーション幅、折り返し |

#### 調整が必要なCSSプロパティ

```css
/* 英文版用のスタイル調整例 */
/* 方法1: クラスで切り替え */
.lang-en {
  font-size: 0.9em; /* 英文は少し小さめに */
  line-height: 1.7; /* 行間を広めに */
}

.lang-en .proto-btn {
  padding: 10px 16px; /* ボタンのパディングを調整 */
  min-width: auto; /* 最小幅を調整 */
}

.lang-en .proto-header nav a {
  padding: var(--spacing-sm) var(--spacing-sm); /* ナビゲーションのパディングを調整 */
  font-size: 0.9em; /* フォントサイズを調整 */
}

/* 方法2: 属性セレクタで切り替え */
html[lang="en"] {
  /* 英文版のスタイル */
}

html[lang="ja"] {
  /* 和文版のスタイル */
}
```

### 2.3.3 ファイル命名規則

英文版のプロトタイプは、以下の命名規則で作成します：

```
prototype_SCREEN-000_homepage_v1.0_en.html  (英文版)
prototype_SCREEN-000_homepage_v1.0.html     (和文版、デフォルト)
```

または、言語をディレクトリで分ける：

```
interactive/
├── ja/
│   └── prototype_SCREEN-000_homepage_v1.0.html
└── en/
    └── prototype_SCREEN-000_homepage_v1.0.html
```

### 2.3.5 確認項目

英文版作成後、以下の項目を確認します：

- [ ] **レイアウトの崩れ**: テキストが長くなってレイアウトが崩れていないか
- [ ] **フォントサイズ**: 英文でも読みやすいフォントサイズか
- [ ] **マージン・パディング**: テキストの長さに応じて適切な余白があるか
- [ ] **ボタン幅**: ボタンのテキストが収まっているか
- [ ] **ナビゲーション**: ナビゲーションメニューが折り返さないか
- [ ] **カード・コンテナ**: カード内のテキストが適切に表示されているか
- [ ] **フォーム**: フォームのラベルと入力欄の配置が適切か

### 2.3.6 実装段階での対応

プロトタイプ段階では、**和文版と英文版を別ファイルで作成**しますが、実装段階では以下のように対応します：

- **i18nライブラリの使用**: `react-i18next`、`next-i18next`等を使用
- **言語切り替え機能**: ユーザーが言語を切り替えられる機能を実装
- **動的な言語切り替え**: ページリロードなしで言語を切り替え

**プロトタイプでは**: 別ファイルで作成（`_en.html`等）  
**実装段階では**: i18nライブラリで動的に切り替え

### 2.3.7 まとめ

| 項目 | プロトタイプ段階 | 実装段階 |
|------|----------------|---------|
| **対応方法** | 和文版と英文版を別ファイルで作成 | i18nライブラリで動的に切り替え |
| **確認内容** | レイアウト、フォントサイズ、マージンの調整 | 言語切り替え機能、翻訳ファイルの管理 |
| **実施タイミング** | 和文版作成後、一括で英文版を作成 | 実装段階でi18nライブラリを統合 |

**結論**: 
- **プロトタイプ段階**: 和文版を作成後、一括で英文版を作成し、レイアウト・フォントサイズを調整
- **実装段階**: i18nライブラリを使用して動的な言語切り替えを実装

## 3. プロトタイプ段階で追加**しない**機能（実装段階に回す）

### 3.1 ❌ サーバー連携が必要な機能

| 機能 | 理由 | 実装段階での対応 |
|------|------|----------------|
| **認証・認可** | Firebase Authenticationが必要 | 実装段階でFirebase SDKを統合 |
| **データの永続化** | Firestoreへの保存が必要 | 実装段階でFirestore SDKを統合 |
| **API連携** | バックエンドAPIが必要 | 実装段階でAPIエンドポイントを実装 |
| **ファイルアップロード** | ストレージサービスが必要 | 実装段階でFirebase Storage等を統合 |
| **メール送信** | サーバー側の処理が必要 | 実装段階でFirebase Functions等を実装 |

### 3.2 ❌ 複雑なビジネスロジック

| 機能 | 理由 | 実装段階での対応 |
|------|------|----------------|
| **暗号化・復号化** | セキュリティ要件が複雑 | 実装段階で暗号化ライブラリを統合 |
| **複雑な計算処理** | パフォーマンス要件がある | 実装段階で最適化を考慮 |
| **リアルタイム同期** | WebSocket等の技術が必要 | 実装段階でFirestoreのリアルタイムリスナーを実装 |

### 3.3 ❌ 本番環境でのみ確認できる機能

| 機能 | 理由 | 実装段階での対応 |
|------|------|----------------|
| **パフォーマンス最適化** | 本番データ量で確認が必要 | 実装段階でパフォーマンステストを実施 |
| **エラーハンドリング** | 実際のエラーケースが必要 | 実装段階でエラーケースを網羅的にテスト |
| **セキュリティテスト** | 実際の攻撃シナリオが必要 | 実装段階でセキュリティテストを実施 |

## 4. プロトタイプ段階での実装レベル

### 4.1 実装レベル別の分類

#### レベル1: 静的表示のみ（現在の状態）
- HTML/CSSのみ
- インタラクションなし
- **例**: 現在のホームページの一部セクション

#### レベル2: 基本的なインタラクション（推奨）
- クリック、ホバー、フォーカス等の基本的なイベント
- 表示/非表示の切り替え
- タブ、アコーディオン等のUIコンポーネント
- **例**: マイページのタブ切り替え（既に実装済み）

#### レベル3: モックデータを使った機能（推奨）
- ローカルデータ（配列、オブジェクト）を使った一覧表示
- 検索・フィルタリング・ソート
- フォームバリデーション
- **例**: PDCAエントリの一覧表示（モックデータ）

#### レベル4: ローカルストレージを使った状態管理（任意）
- `localStorage`を使った一時保存
- 表示状態の保持
- **例**: フォーム入力の下書き保存（モック）

#### レベル5: サーバー連携（実装段階）
- Firebase Authentication
- Firestoreへの保存
- API連携
- **実装段階で対応**

## 5. 具体的な実装例

### 5.1 ✅ 推奨：追加できる機能の実装例

#### 例1: フォームバリデーション（お問い合わせフォーム）

```javascript
// お問い合わせフォームのバリデーション
document.querySelector('form').addEventListener('submit', function(e) {
  e.preventDefault(); // サーバー送信はしない
  
  const name = document.getElementById('contact-name').value;
  const email = document.getElementById('contact-email').value;
  const subject = document.getElementById('contact-subject').value;
  const message = document.getElementById('contact-message').value;
  
  // バリデーション
  if (!name || !email || !subject || !message) {
    alert('すべての項目を入力してください');
    return;
  }
  
  // メール形式チェック
  const emailRegex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
  if (!emailRegex.test(email)) {
    alert('正しいメールアドレスを入力してください');
    return;
```

## 6. プロトタイプの更新ルール

### 6.1 バージョン管理

- プロトタイプのバージョンは `v1.0`, `v1.1` 等で管理
- 大きな機能追加時は `v2.0` にバージョンアップ
- ファイル名にバージョンを含める: `prototype_SCREEN-000_homepage_v1.0.html`

### 6.2 機能追加時の注意点

1. **モックであることを明示**: プロトタイプでは実際の処理を行わないことを明記
   - `proto-mock-note` クラスを使用
   - `alert()` やコメントでモックであることを表示

2. **実装段階への引き継ぎ**: プロトタイプで実装した機能は、実装段階で参考にする
   - コメントで実装方針を記載
   - モックデータの構造を実データ構造に合わせる

3. **パフォーマンスを考慮しない**: プロトタイプではパフォーマンス最適化は不要
   - モックデータは少量で問題なし
   - 最適化は実装段階で実施

### 6.3 直接修正時の確認タイミング

プロトタイプの細かい修正は手動で行うことが推奨されますが、以下のタイミングで確認・レビューを行うことを推奨します。

#### 6.3.1 確認が必要なタイミング

| タイミング | 確認内容 | 確認方法 |
|----------|---------|---------|
| **機能追加時** | 新しい機能を追加した後 | 動作確認、レイアウト確認 |
| **大きな変更時** | レイアウトの大幅な変更、新しい画面の追加 | 全体の整合性確認 |
| **バージョンアップ時** | `v1.0` → `v1.1` 等のバージョン変更時 | 変更内容の確認 |
| **実装段階前** | 実装段階に移行する前 | 最終確認、実装方針の確認 |

#### 6.3.2 確認が不要なタイミング

| 修正内容 | 理由 |
|---------|------|
| **細かいテキスト修正** | 誤字脱字、表現の微調整等 | 手動で修正して問題なし |
| **CSSの微調整** | マージン、パディング、色の微調整等 | 手動で修正して問題なし |
| **リンクの追加・修正** | ナビゲーションリンクの追加等 | 手動で修正して問題なし |
| **スタイルの調整** | フォントサイズ、ボタンサイズ等の調整 | 手動で修正して問題なし |

#### 6.3.3 推奨されるワークフロー

```
1. 細かい修正を手動で実施
   ↓
   テキスト修正、CSS調整、リンク追加等
   ↓
2. ローカルで動作確認
   ↓
   ブラウザで実際に動作を確認
   ↓
3. 大きな変更・機能追加時のみ確認依頼
   ↓
   新しい機能の追加、レイアウトの大幅な変更等
   ↓
4. バージョンアップ時に確認
   ↓
   変更内容をまとめて確認
```

#### 6.3.4 確認依頼時の情報提供

確認を依頼する際は、以下の情報を提供するとスムーズです：

- **変更内容**: 何を変更したか（例: 「ホームボタンを追加」）
- **変更したファイル**: どのファイルを変更したか
- **確認してほしい点**: 特に確認してほしい箇所（例: 「レスポンシブ対応が正しいか」）
- **動作確認済みか**: ローカルで動作確認済みかどうか

#### 6.3.5 まとめ

**手動で修正して問題ない項目**:
- テキストの修正（誤字脱字、表現の調整）
- CSSの微調整（マージン、パディング、色等）
- リンクの追加・修正
- スタイルの調整（フォントサイズ、ボタンサイズ等）

**確認を依頼した方が良いタイミング**:
- 新しい機能を追加した時
- レイアウトを大幅に変更した時
- バージョンアップ時（`v1.0` → `v1.1`等）
- 実装段階に移行する前の最終確認

**結論**: 
- **細かい修正**: 手動で修正して問題なし。都度確認は不要。
- **大きな変更・機能追加**: 確認依頼を推奨。
- **バージョンアップ時**: 変更内容をまとめて確認。

## 7. チェックリスト

### プロトタイプに追加する機能を検討する際のチェックリスト

- [ ] **サーバー連携は不要か？**
  - ✅ クライアントサイドのみで完結する機能 → 追加可能
  - ❌ サーバー側の処理が必要 → 実装段階に回す

- [ ] **UI/UXの確認に必要か？**
  - ✅ レイアウト、操作性の確認に必要 → 追加推奨
  - ❌ ビジネスロジックのみ → 実装段階に回す

- [ ] **モックデータで代替可能か？**
  - ✅ モックデータで動作確認できる → 追加可能
  - ❌ 実際のデータが必要 → 実装段階に回す

- [ ] **実装コストは適切か？**
  - ✅ 数時間〜1日程度で実装可能 → 追加推奨
  - ❌ 数日以上かかる → 実装段階に回す

## 8. まとめ

### プロトタイプ段階で追加・修正できる機能

✅ **推奨**:
- UI/UX確認に必要な基本的なインタラクション
- モックデータを使った一覧表示、検索、フィルタリング
- フォームバリデーション（クライアントサイド）
- モーダル、タブ、アコーディオン等のUIコンポーネント
- ローカルストレージを使った簡易的な状態管理（任意）

❌ **実装段階に回す**:
- サーバー連携（認証、データ保存、API連携）
- データの永続化（Firestore等）
- 複雑なビジネスロジック
- 本番環境でのみ確認できる機能

### 判断基準

**「UI/UXの確認に必要か？」** が最も重要な判断基準です。
- UI/UXの確認に必要 → プロトタイプで追加
- ビジネスロジックのみ → 実装段階に回す
