## 0. ユーザーの状態（認証前・認証後）

プログラムなどのサービスを利用する**前**の段階と、**ログイン後**の段階を分けて定義する。

### 0.1 未ログインの訪問者（ゲスト）

| 項目 | 内容 |
|------|------|
| **誰か** | ホーム画面の動画・リンク・記事を**読むだけ**の人。まだプログラムやマイページは利用していない。 |
| **認証** | **ログインしていない**。Firestore にユーザードキュメントは存在しない（またはログイン後に初めて作成される）。 |
| **できること** | ホームの公開コンテンツ（最新動画、参考リンク、記事など）の**閲覧**。 |
| **ログインが必要になるタイミング** | **何か無料サービスを実行しようとしたとき**（例: 28日間こころのトライアルを「始める」、マイページを見る、プログラムに記入する）。その時点で Google ログインを求める。 |

- ゲストは「ロール」を持たない。DB 上は「未認証」である。
- 一般ユーザ（クライアント）は、**ログインしたあと**の役割である。

### 0.2 ログイン済みユーザ（クライアント・コーチ・管理者）

| 項目 | 内容 |
|------|------|
| **誰か** | Google でログインし、アカウント（ユーザープロファイル）が存在する人。 |
| **できること** | ロール（下記 1.1）に応じて、プログラムの実施・マイページ・コーチとの共有・管理機能などを利用する。 |
| **一般ユーザ＝クライアント** | プログラムを**利用する側**。28日間トライアルの記入、マイページの閲覧・編集、コーチとの情報共有など。 |

- **ロール**（client / coach / admin）は、**ログイン済みユーザ**にのみ付与される。未ログインの訪問者にはロールはない。

### 0.3 整理

| 状態 | 説明 | ロール | 例 |
|------|------|--------|-----|
| **未ログイン（ゲスト）** | ホームの動画・リンク・記事を見る人。無料サービスを「実行」する前。 | なし | サイトを初めて訪れた人 |
| **ログイン済み・一般ユーザ** | 無料サービス（トライアル等）を始めるためにログインした人。プログラムを実施する。 | `client`（または `user`） | 28日間トライアルに記入する人 |
| **ログイン済み・ホスト** | クライアントのサポート・管理を行う人。 | `coach` | コーチ |
| **ログイン済み・管理者** | サイト全体のコンテンツを登録・編集する人。 | `admin` | アドミニストレータ |

## 1. ユーザー権限（ロール）

※ 以下は**ログイン済みユーザ**に付与されるロールである。

### 1.1 ロール一覧

| 種別（表示名） | ロールID | 役割 | 閲覧・編集権限 |
|----------------|----------|------|----------------|
| **一般ユーザ** | `user` | クライアント | 各種プログラムの実施、マイページの閲覧・記入・編集、コーチとの情報共有 |
| **ホストユーザ** | `coach` | コーチ | クライアントとの情報共有、クライアントの管理 |
| **管理者** | `admin` | アドミニストレータ | 最新動画・参考リンクの設定など、サイト全体の登録・編集 |

- **一般ユーザ（クライアント）** は、**ログインしたあと**に付与されるロール。未ログインの訪問者（ゲスト）はロールを持たない（上記「0. ユーザーの状態」を参照）。
- 実装上のロール値は **`user`** | **`coach`** | **`admin`** の3種とする（ロール名は `user` のまま）。
- 既存コードに `senior_coach` がある場合は、当面未使用とするか `coach` に統合する。

### 1.2 権限の使い分け

- **user（一般ユーザ）**: ホーム、マイページ、28日間トライアル、自己理解・目標設定・PDCA 等のプログラム利用。共有設定に応じてコーチと共有可能。
- **coach（ホストユーザ）**: 担当クライアント一覧、クライアントが「共有OK」にした範囲の閲覧。自分がコーチであることと、画面内のどこが共有されているかを判別できる必要がある。
- **admin（管理者）**: ホームの「最新動画」「参考リンク」等のサイト共通コンテンツの登録・編集、コーチの付与（DB で設定）。**他ユーザの記録は IF 上からは閲覧できない**（後述 1.5）。

### 1.3 データ・画面アクセス（概要）

| 対象 | user | coach | admin |
|------|------|-------|-------|
| マイページ（自分のみ） | ✅ | ✅（自分のみ） | ✅（自分のみ） |
| プログラム（トライアル等）の実施 | ✅ | ✅ | ✅ |
| 他ユーザーの記録閲覧（UI） | ❌ | ✅（共有OKの範囲のみ） | ❌（IF 上からは見れない） |
| サイト共通コンテンツの編集 | ❌ | ❌ | ✅ |

#### 1.3.1 アファメーション（補足）

| ロール | 内容 |
|--------|------|
| **user（クライアント）** | 自身のアファメーションの作成・下書き・発行・編集・削除・履歴（仕様どおり）。 |
| **coach** | **発行済みかつクライアントが共有可能にしたアファメーション** の本文を閲覧。下書き・本文編集・削除・編集履歴（`history`）は行わない。**A-11** で `coach_comment_versions` への追記・担当は `coach_client_assignments`（[03_A11_COACH_SHARING_SCHEMA_DRAFT.md](./03_A11_COACH_SHARING_SCHEMA_DRAFT.md)）。画面遷移は [04_AFFIRMATION_DESIGN.md](./04_AFFIRMATION_DESIGN.md) 3.7・3.10。 |
| **admin** | ユーザーのアファメーション本文を **IF から閲覧しない**（既存 1.5 と同方針）。システム用プロファイル `affirmation_profiles` の編集は管理者権限で可（ルール実装時）。 |

### 1.4 URL とロールの考え方

- **URL は画面（機能）単位で共通**とする。ロールごとに URL を変えない。
- 同じ URL でも、**ロールと「共有設定」**によって表示内容・操作可能範囲を変える。
  - 例: `/trial_4w` は 1 つの URL。user は自分の記入と「この項目をコーチと共有する」などの設定ができ、coach は「共有OK」の項目だけが見える。画面のどこが共有対象かが分かるようにする。

### 1.5 ロール付与・管理者の閲覧制限

| 項目 | 方針 |
|------|------|
| **ロールの付与** | DB（Firestore の `users/{uid}.role`）で設定する。 |
| **コーチの付与** | **管理者（admin）** が、DB 上で該当ユーザの `role` を `coach` に設定する（管理画面から行うか、手動で Firestore を更新するかは別途決める）。 |
| **管理者による他ユーザ記録の閲覧** | **IF（画面）からは行えない**ようにする。DB には暗号化等で保存されているため内容は直接は見えないが、管理者用の「他ユーザ記録閲覧」画面は提供しない。 |
| **同一人物が admin と coach** | 可。その場合はコーチとして「共有OK」のクライアント記録を閲覧できる。管理者としての「全記録閲覧」機能は持たない。 |

### 1.6 表示モードの切替（クライアント／ホスト／管理者）

ホスト（coach）や管理者（admin）も、**自分自身がクライアントとして**プログラム・マイページを利用したい場合がある。そのため「いまどの立場で画面を見るか」を**モード**で切り替える設計とする。

#### 方針

| 項目 | 内容 |
|------|------|
| **デフォルトの見え方** | ログイン後は**クライアントモード**（自分用のホーム・トライアル・マイページなど）がデフォルト。ホスト・管理者も同じ。 |
| **ホストモード** | **ホスト権限（coach）を持つ人だけ**が切り替え可能。ホストモードにすると、担当クライアント一覧・共有された記録など、コーチ用の画面が表示される。 |
| **管理者モード** | **管理者権限（admin）を持つ人だけ**が切り替え可能。管理者モードにすると、最新動画・参考リンクの編集など、サイト管理用の画面が表示される。 |
| **モードの種類** | 「クライアントモード」「ホストモード」「管理者モード」の3種。ユーザーは**持っている権限に応じたモードだけ**切り替えられる。 |

#### 誰がどのモードを使えるか

| ロール（DB） | クライアントモード | ホストモード | 管理者モード |
|--------------|-------------------|--------------|--------------|
| **user のみ** | ✅ 常にこの見え方 | ❌ 使えない | ❌ 使えない |
| **coach** | ✅ デフォルト。自分用に利用 | ✅ 切替可能 | ❌ 使えない |
| **admin** | ✅ デフォルト。自分用に利用 | ❌ 使えない | ✅ 切替可能 |
| **coach かつ admin**（複数ロールの場合） | ✅ デフォルト | ✅ 切替可能 | ✅ 切替可能 |

- 複数ロールを同一人物に付与する場合は、DB で `role` を配列やフラグで持つか、別テーブルで「この uid は coach かつ admin」と管理する。実装方針は別途決める。

#### 成り立つ理由（まとめ）

- **ホスト・管理者もクライアントとして利用したい** → 普段はクライアントモードで、自分用のコンテンツだけ見える。
- **ホスト／管理者の仕事をするときだけ** → モードを切り替えて、それぞれの専用画面にアクセスする。
- **モード切替が使えるのは権限を持つ人だけ** → role が coach でないとホストモードは選べない。admin でないと管理者モードは選べない。
- これにより「一人の人がクライアントとしての利用」と「ホスト／管理者としての利用」を同じアカウントで使い分けられる。

#### モードの保持（セッション以外でも利用）

- モードは**セッションだけ**に限定しない。次回ログイン後も「前回ホストモードで見ていたクライアント」や「デフォルトでホストモードで開く」など、利用シーンに合わせて保持する想定とする。
- 例: ユーザープロファイルに「最後に使った表示モード」「ホストモードで最後に選択したクライアントID」を保存する、またはブラウザのローカルストレージで保持する。画面設計・実装方針でどれをどこまで保持するか決める。

#### ホストモードでのやり取りフロー（コレスポンデンス）

コーチがクライアントの共有コンテンツに対してコメントするなど、**コレスポンデンス**を想定する。おおよその流れは次のとおり。

| 順序 | ステップ | 説明 |
|------|----------|------|
| 1 | **モード切替** | クライアントモード → ホストモードに切り替える。 |
| 2 | **クライアント指定** | 担当クライアント一覧から、対象のクライアントを選ぶ。 |
| 3 | **クライアント共有コンテンツ表示** | そのクライアントが「コーチと共有」にしたコンテンツ（例: 週報、28日間トライアルの共有項目）を表示する。 |
| 4 | **コメント記入** | 共有されている項目ごとにコーチがコメントを記入・保存する。クライアントは後から自分の画面でそのコメントを確認できる。 |

- **アファメーション（A-11）** のデータ置き場・フィールド名の現状案は [03_A11_COACH_SHARING_SCHEMA_DRAFT.md](./03_A11_COACH_SHARING_SCHEMA_DRAFT.md)（割当 `coach_client_assignments`、`coach_share_rounds` / `coach_comment_versions`）。
- **マネジメント日誌（学び帳）** の人コーチ／AI／サブスク **コース定義・能力マトリックス** は [03_JOURNAL_COACH_AI_PLANS_AND_CAPABILITIES.md](./03_JOURNAL_COACH_AI_PLANS_AND_CAPABILITIES.md)。日次は人コーチに共有しない。週次は共有のみ、月次は質問・回答含む（現行プラン `ai_plus_personal`）。動的ルールではなく **コース（運用）で切替**。
- この流れを**画面設計で考慮**し、ホーム・28日間トライアルそれぞれで「モード切替時の見え方」と「クライアントとの情報共有（コメントのやり取り）」のイメージを用意する（後述「3. 画面設計で作成するイメージ・成果物」）。

## 2. 画面単位の権限と共有設定

### 2.1 URL は画面単位・中身でロール別に出し分け

- **ロールごとに URL を変えない**。画面（機能）ごとに 1 つの URL を割り当て、その画面内で「誰が何を見れるか・何を設定できるか」をロールと共有設定で制御する。
- コーチは「自分がコーチであること」を画面上で認識できる必要がある（例: コーチ用メニューや「担当クライアント」エリアの表示）。
- また、各画面で「どこが共有されているか」が分かるようにする（例: 28日間トライアルで「この項目はコーチと共有中」などの表示）。

### 2.2 共有設定（ visibility ）のイメージ

ユーザが記入した内容について、「誰まで見せるか」を選べるようにする。SNS の「自分のみ／友達のみ／公開」に近い考え方。

| 設定値（例） | 説明 | コーチの閲覧 |
|--------------|------|----------------|
| **自分のみ** | 本人のみ閲覧・編集 | ❌ 見れない |
| **コーチと共有** | 担当コーチが閲覧可能 | ✅ 見れる（共有OKの部分のみ） |
| **公開** | （将来用）より広い公開範囲 | 仕様次第 |

- 各画面・各データ種別ごとに、上記のどれをサポートするかは設計で決める（例: 28日間トライアルの「朝の記録」は「自分のみ」「コーチと共有」の 2 段階など）。
- コーチ側の画面では、「共有OK」になっている項目だけが一覧・詳細に表示され、共有されていない部分は表示しない（または「非共有」であることが分かる表示にする）。

### 2.3 画面 × アクセス権 一覧表

画面（URL）ごとに、ゲスト・user・coach・admin のアクセスと、その画面で扱う「共有設定」を整理する。

| 画面（URL） | ゲスト | user | coach | admin | 共有設定の扱い（その画面で） |
|-------------|--------|------|-------|-------|------------------------------|
| **`/`（ホーム）** | ✅ 閲覧のみ（動画・リンク・記事） | ✅ 同上＋ログイン後用ブロック表示 | ✅ 同上 | ✅ 同上＋管理用リンク等 | なし（公開コンテンツのみ） |
| **`/trial_4w`（28日間トライアル）** | ❌ 閲覧のみ or ログイン促し（方針次第） | ✅ 自分の記入・編集。項目ごとに「コーチと共有」ON/OFF 可能 | ✅ 担当クライアントの「共有OK」項目のみ閲覧。どこが共有か分かる表示 | ✅ 自分の記入のみ（他ユーザ記録は見ない） | 自分のみ / コーチと共有 |
| **`/mypage`** | ❌ ログイン必須 | ✅ 自分のマイページ閲覧・編集 | ✅ 自分のマイページのみ | ✅ 自分のマイページのみ | 必要に応じて検討 |
| **`/login`** | ✅ ログイン画面表示 | ✅ すでにログイン済みならリダイレクト等 | 同上 | 同上 | なし |
| **コーチ用（例: `/coach` または `/mypage` 内）** | ❌ | ❌ または 非表示 | ✅ 担当クライアント一覧・共有OK項目の閲覧 | ❌ または 管理者は別メニューのみ | コーチは「共有OK」の範囲のみ表示 |
| **管理用（例: `/admin`）** | ❌ | ❌ | ❌ | ✅ サイト共通コンテンツの登録・編集、コーチ付与等。**他ユーザ記録の閲覧は提供しない** | なし |

- 上記は現時点の想定。実際の URL（例: コーチ用を `/coach` にするか `/mypage` のタブにするか）は画面設計に合わせて調整する。
- 「他ユーザの記録」は、admin には **IF 上からは見せない**。コーチには「共有OK」にした部分だけを見せる。

### 2.4 コーチが認識すべきこと（画面設計でのポイント）

- **自分がコーチであること**: ログイン後、`role === 'coach'`（または coach を含む）の場合に、コーチ用のメニュー・ラベル・エリア（例: 「担当クライアント」「共有されている記録」）を表示する。
- **どこが共有されているか**: クライアントの記録一覧・詳細で、「この項目は共有」「この項目は非共有」が分かる表示（アイコン・ラベル・フィルタなど）を検討する。

### 2.5 ホーム画面の content-section ID 一覧（ロール別表示用）

ロール別に項目の表示を区別するため、ホームの各セクションに識別用 ID を付与している。**ロール別の詳細な区分（表示・非表示・編集可）** と **管理者モード時の編集UI・参考リンクのテーブル形式編集画面・実装前の確認事項・ステップ** は [04_HOME_SCREEN_IMPLEMENTATION.md](./04_HOME_SCREEN_IMPLEMENTATION.md) にまとめた。

| ID | セクション名 | ゲスト | クライアント/ホスト | 管理者 | 備考 |
|----|----------------|--------|---------------------|--------|------|
| `home-banner` | バナー | 〇 | 〇 | 〇 | 文言はログイン前/後で切替 |
| `home-section-latest-videos` | 最新動画 | 〇 | 〇 | ◎ | 管理者のみ編集可 |
| `home-section-today-best` | 本日の一番 | - | 〇 | 〇 | ゲストは非表示 |
| `home-section-continuation` | 昨日までの積重ね | - | 〇 | 〇 | ゲストは非表示 |
| `home-section-latest-articles` | 最新記事 | 〇 | 〇 | ◎ | 管理者のみ編集可 |
| `home-section-sns` | SNS | 〇 | 〇 | 〇 | 表示のみ |
| `home-section-reference-links` | 参考リンク | 〇 | 〇 | ◎ | 管理者のみ編集可（テーブル形式でURL入力） |
| `home-section-ad` | 広告エリア | 〇 | - | ◎ | クライアント・ホストは非表示。管理者のみ編集可 |

- 凡例: 〇=表示、◎=表示＋編集可、-=非表示。
- 実装: `src/components/HomePage.tsx` の各 `<section>` に `id` を付与済み。

## 3. 画面設計で作成するイメージ・成果物

モード切替とクライアントとの情報共有（コレスポンデンス）を画面で実現するため、**画面設計時**に次のイメージ・成果物を作成することを推奨する。

### 3.1 モード切替時の画面イメージ

各画面で「クライアントモード」「ホストモード」「管理者モード」の**見え方の違い**を図やワイヤーフレームで整理する。

| 対象画面 | 作成するイメージ | 用途 |
|----------|------------------|------|
| **ホーム画面（`/`）** | ・クライアントモード時のレイアウト・表示項目<br>・ホストモード時のレイアウト（例: 担当クライアント一覧への入口、モード切替UI）<br>・管理者モード時のレイアウト（例: 最新動画・参考リンク編集への入口、モード切替UI） | 実装・レビュー時の共通認識 |
| **28日間トライアル画面（`/trial_4w`）** | ・クライアントモード時: 自分の記入・共有ON/OFF設定<br>・ホストモード時: クライアント指定→そのクライアントの共有コンテンツ表示→コメント記入の流れが分かる画面構成 | 同上。ホストモードでの「週報へのコメント」などのやり取りを想定 |

- モード切替の**トリガー**（ヘッダーのドロップダウン、サイドバーのスイッチなど）がどの画面のどこに現れるかも、イメージに含めるとよい。

### 3.2 クライアントとの情報共有イメージ（コレスポンデンス）

ホストモードでの**コーチ⇔クライアント**のやり取りを、画面の流れとして図示する。

| 成果物 | 内容 |
|--------|------|
| **フロー図・画面遷移** | 1. モード切替（→ホストモード）→ 2. クライアント指定（一覧から選択）→ 3. クライアント共有コンテンツ表示（週報・28日間の共有項目など）→ 4. コメント記入・保存。必要に応じて「クライアントがコメントを確認する画面」も含める。 |
| **画面イメージ（コーチ側）** | 共有コンテンツ一覧、項目ごとの詳細表示、コメント入力欄・送信の配置。 |
| **画面イメージ（クライアント側）** | 自分の記入内容と、コーチからのコメントがどこに・どう表示されるか（例: 週報の該当行の横にコメント表示）。 |

- 上記を用意しておくことで、ホーム・28日間画面それぞれで「モード切替時」「共有コンテンツ表示」「コメント記入」を一貫した体験として設計できる。

## 4. サブスクリプション（技術的器）

### 4.1 方針

- **サブスク形式**でサービスを提供する。
- **決済・課金は本フェーズでは実装しない**。プラン種別・トライアル期間・有効期限などを保持する**技術的な器**を先に用意する。
- 商品（フリー、期間限定、コース、個別サービス追加など）の詳細は、サービス提供内容に合わせて後から決める。

### 4.2 想定するプラン種別（器として）

| 種別 | 説明（例） |
|------|------------|
| **free** | 無料。利用範囲は別途定義。 |
| **trial** | トライアル中（例: 28日間こころのトライアルの無料期間）。終了日を持ち、以降は有料プランへ誘導。 |
| **course** | コース契約（例: 月額・年額）。商品ごとに詳細を定義。 |
| **addon** | 個別サービス追加。コースに付加するオプション等。 |

- 上記は**データ構造・列挙として用意**し、実際の「どのプランで何ができるか」はプロダクト要件に合わせて後から詰める。

### 4.3 28日間こころのトライアルの扱い

- **28日間はフリー**で利用可能。
- **28日経過後は有料**（プラン変更または制限付き継続）。
- 技術的には次のいずれかで表現する：
  - プラン `trial` で `trialEndDate`（開始日＋28日）を持たせる。
  - または `plan: 'free'` のまま `trialEndDate` のみ持ち、トライアル終了後に `plan` を `course` 等に更新する（決済実装時）。
- 現段階では **トライアル終了日とプラン種別を保持するフィールド** をユーザープロファイル（または subscription サブオブジェクト）に用意し、画面では「トライアル残り日数」「期限後は有料」などの表示ができるようにする。

### 4.4 保持するデータ（器）

- **プラン種別**（例: `free` | `trial` | `course` | `addon` 等、商品に合わせて拡張）
- **ステータス**（例: `active` | `inactive` | `expired` | `cancelled`）
- **開始日・終了日**（`startDate`, `endDate`）
- **トライアル終了日**（`trialEndDate`）— 28日間トライアル用
- **機能フラグ・利用制限**（既存の `features` / `usage` と同様の構造を維持・拡張）

決済ID・課金履歴は**決済実装フェーズ**で追加する。
