## 1. アファメーションのプロファイル（穴埋めテンプレート）

### 1.1 要件

- アファメーションの入力は **穴埋め式** とする。
- 穴埋めの「どこに何を入れるか」は **プロファイル** で定義する。
- 将来的にユーザーが **プロファイルの作成・編集・選択** ができる構造にする。
- **今回** は **指定のプロファイル 1 つのみ** を使用し、選択機能は実装しない。

### 1.2 プロファイルの構造案

1 プロファイル = 複数の「見出し」を持ち、各見出しの下に「文字列」と「入力フィールド（穴）」を交互に並べる。

| 項目 | 説明 | 例 |
|------|------|-----|
| **プロファイル名** | プロファイルの識別・表示用 | `アファメーション01` |
| **見出し** | セクションのラベル | `一年後の自分`、`行動宣言（Do）` |
| **文字列** | 穴の前後の固定テキスト | `私は、`、`までに、`、`そして、` |
| **入力フィールド幅** | 穴の目安文字数（表示・バリデーション用） | `150` |

**例（1プロファイル内の流れ）**

```
プロファイル名: アファメーション01

【見出し】一年後の自分
  文字列: 「私は、」
  入力フィールド幅: 150
  文字列: 「までに、」
  入力フィールド幅: 150
  文字列: 「そして、」
  入力フィールド幅: 150

【見出し】行動宣言（Do）
  文字列: 「私は、この状態を実現するために、」
  入力フィールド幅: 150
  …
```

- 各「穴」には **一意の ID（スロットID）** を付与し、後述の「入力データ」と紐づける。
- プロファイルの作成・編集ができるタイミング（例: 設定画面、プログラム開始前など）は別途検討。今回は **固定 1 プロファイル** のみ利用。

### 1.3 プロファイルの保管場所（確定）

- **テンプレート（プロファイル）の正本は Firestore に保存する**（将来の「テンプレート作成・編集 UI」との整合のため。サーバ上の単一テキストファイルのみに依存しない）。
- **システム／管理者定義**: ルートコレクション **`affirmation_profiles/{profileId}`**（`site_content/home` が単一ドキュメントのため物理パスは分離。論理的にはサイト共通マスタ。[03_FIRESTORE_DATABASE_STRUCTURE.md](./03_FIRESTORE_DATABASE_STRUCTURE.md) 2.10 参照）。
- **ユーザー定義・ユーザー改変**: `users/{uid}/affirmation_profiles/{profileId}`。
- **今回のプロト実装**: 開発中は `src/lib/affirmationProfile.ts` の定数（例: `AFFIRMATION_PROFILE_V1`）と併用しつつ、本番データ連携時は上記 Firestore と同期する想定。
- **発行後**: 完成文書は **プロファイルから切り離された 1 本文** として保存する（後述 `users/{uid}/affirmations/{affirmationId}`）。プロファイルは参照用メタ（元 `profileId`）のみ残すか、運用で決める。

### 1.4 発行時に生成する Markdown の記法（確定）

**マスタデータは §1.5 の `sections` / `blocks` を正とする**（方式の決定は §1.6）。発行処理では、ブロック列から **機械的に** 次の記法を満たす **1 本の Markdown 文字列** を組み立て、スロットをユーザー入力で置換したうえで Firestore に保存する（完成文は §3）。

| 記法 | 意味 |
|------|------|
| `[[slotId:最大文字数]]` | `slotId` に対応する穴。例: `[[future.self:50]]` は `future.self` スロット・最大 50 文字。 |
| 改行 | ブロック生成ルールに従い、発行 Markdown に反映する。 |
| リスト・見出し | 通常の Markdown（`-` リスト、`##` 見出し等）に準拠。**見出しブロックは `heading` 等から生成**する。 |

- **理由**: `[50]` のみではスロットと他プログラム入力の **ID 紐づけ** ができないため、生成 Markdown では **必ず `slotId` を含める**（`[[slotId:length]]`）。
- **制作者があらかじめ Markdown だけを正として編集する方式（markdown テンプレート単体をマスタにしない）** としない。Markdown は **発行パイプラインの出力形式**。
- 表示側は Markdown レンダラで表示する（アプリ内プレビュー・発行済み閲覧で共通方針）。実装: **`AffirmationMarkdownView`**（`src/components/common/AffirmationMarkdownView.tsx`、`react-markdown`）。

### 1.5 プロファイル 1 件のスキーマ（Firestore／アプリ内表現の対応）

**正本**: `sections` と各 `blocks` の列（方式の決定は §1.6）。

```ts
// 穴 1 つ分
{ slotId: string, label?: string, maxLength: number }  // 後ろの「文字列」は別フィールドで並びで管理

// 見出し + 並び付きブロック列
section: {
  heading: string;           // 「一年後の自分」
  blocks: Array<
    | { type: 'text'; text: string }
    | { type: 'slot'; slotId: string; maxLength: number }
  >;
}

// プロファイル 1 件
{
  id: string;
  name: string;              // 「アファメーション01」
  sections: Array<{ heading: string; blocks: [...] }>;
  createdAt, updatedAt
  markdownTemplate?: string;  // 任意・非正本: blocks から生成した Markdown のコピー（エクスポート・検証用）。編集のマスタにしない。
}
```

### 1.6 ブロック式を正とする（決定：2026-03-02）

| 方式 | 内容 |
|------|------|
| **ブロック式（採用）** | プロファイルは **見出し＋`blocks`（固定テキスト／スロット）のシリアル**として保持する。UI はこの順で並べればよい。**別プログラムでセクションやスロット単位の入力を作り、最後にまとめて表示・発行する**拡張に向く（§2 と整合）。 |
| **Markdown テンプレート単体をマスタにする方式（不採用）** | 制作者があらかじめ穴あき Markdown を 1 本用意し、それを DB の正とする方式。発行前は **テンプレ文字列＋スロット値**中心のイメージになる。 |

- **`markdownTemplate` フィールド**（§1.5 のオプション）は **正本ではない**。必要なら **blocks からのエクスポート・キャッシュ・デバッグ用**に限り利用してよい。

## 2. 入力データの ID 付与と紐づけ

### 2.1 要件

- **プロファイル** と、そのプロファイルから **生成された入力データ** は **ID で紐づける**。
- アファメーションの「穴」の値は、**別プログラム（別ステップ）** で作成されることがある。
  - 例: 「一年後の自分」は目標・計画設定のステップで作成、「行動」は行動策定のステップで作成。
- プログラムのプロセスが一通り終わった段階で、**複数ステップの結果をまとめて 1 つのアファメーション** にする。
- そのため **各入力データ** と **プロファイル内のどの穴（スロット）・どの文字列** に対応するかの **紐づけ** が必要。

### 2.2 紐づけの考え方

| 対象 | 役割 |
|------|------|
| **プロファイル** | プロファイルID で識別。どの見出し・どの順で「文字列 / 穴」が並ぶかを定義。各穴は `slotId` で一意。 |
| **入力データ（穴の値）** | 1 つ 1 つに **入力データ ID** を付与。**どのプロファイルのどの slotId に入るか** を参照で持つ（`profileId` + `slotId`）。 |
| **他プログラムとの連携** | 例: 目標設定ステップで作った文言 → その結果を「入力データ」として保存し、`profileId` + `slotId`（例: 「一年後の自分」の 1 つ目）と紐づける。行動策定ステップで作った文言 → 別の `slotId` と紐づける。 |

- **まとめ時**: プロファイルの順序に従い、各 `slotId` に対応する「入力データ」の値を埋め込んで 1 文を組み立てる。
- 入力データは **プログラム・ステップ横断** で参照できるようにする（同じ `profileId` + `slotId` に複数候補がある場合は「どれを採用するか」のルールを別途検討）。

### 2.3 データ構造（案）

- **入力データ 1 件**: `profileId`, `slotId`, `value`（穴に入る文言）, `sourceStep?`（どのステップで作成したか）, `createdAt` など。
- 保存先: 例）`users/{uid}/affirmation_data/{inputDataId}` または `users/{uid}/coaching-program/...` の下に、プロファイル別・スロット別で持つ。
- **「どのプロファイルのどの穴に、どの入力データを使うか」** を保持するマッピング（例: `profileId` → `{ slotId: inputDataId }`）を、ユーザーごと・アファメーション編集中だけ持つイメージ。

## 3. 作成したアファメーションの編集と履歴

### 3.1 要件

- 穴埋めで組み立てたあと、**全体を通して** 言葉のニュアンス・つなぎを修正したくなる。
- **内容そのもの** も修正したくなる。
- そのため、**一旦できあがった文を「書式付きテキスト」として出力** し、それを **ユーザーが実際に使うアファメーション** とする。
- この段階では **個別センテンスの ID 紐づけはなくし、1 つの文章** として扱う。
- この **完成版アファメーション** には **文書としてのタイトル** および **ID** が必要。
- クライアントは **何度もバージョンアップ** するため、**履歴を残す** 必要がある。

### 3.2 流れの整理

1. **編集前**: プロファイルに従い、各穴（slotId）に入力データを埋めて 1 文を組み立てる。この時点では slotId と入力データの紐づけがある。
2. **「まとめ」／「発行」**: 上記を **Markdown 本文**（1 本の文字列、改行・見出しを含む）に変換して保存。これが「アファメーション文書」の **内容**。発行後はプロファイルとは **論理的に切り離し**、以降は文書として編集する。
3. **編集**: ユーザーがこの **完成文** を直接編集（ニュアンス・つなぎ・内容の変更）。編集後は再び **Markdown（または同一の書式付き表現）** として保存。
4. **履歴**: **発行済み文章（published）のみ**を対象に、保存時に「履歴を残す」を選んだ場合のみ履歴エントリを追加する。「履歴を残さない」を選んだ場合は、履歴を増やさず **現在バージョンを上書き**する。

### 3.3 データ構造（確定・保存パス）

- **アファメーション ID**: `affirmationId` は **同一ユーザ（`uid`）内でユニーク**。名称変更時は UI で重複チェック（`affirmationId` は不変、表示名のみ変更する運用でも可。実装で統一）。
- **保存のルート**: **`users/{uid}/affirmations/{affirmationId}`** 配下（ユーザードキュメントのサブコレクション）。

#### 案 B（採用）：親＝メタ、`published` サブ＝発行済み本文

| 場所 | 内容 |
|------|------|
| **親** `affirmations/{affirmationId}` | **メタのみ**（一覧・権限・状態に必要なフィールド）。例: `title`, `status`（`draft` / `published`）, `profileId`, `createdAt`, `updatedAt`。一覧用の **`encryptedLastPreviewText`** はクエリ都合で親に持ってもよい（暗号化は変わらず）。**A-11 共有**: `sharedWithCoach`, `lastSharedWithCoachAt`, `lastSharedBodyFingerprint`, `coachUnreadAfterClientShare`（任意）。 |
| **サブ** `affirmations/{affirmationId}/published/{docId}` | **発行済みの本文**（暗号化）。固定 ID 例: **`current`**（現行の正）。フィールド例: `encryptedBody`, `publishedAt`, `updatedAt`。 |
| **サブ** `affirmations/{affirmationId}/history/{historyId}` | 発行済みの編集で「履歴を残す」ときのスナップショット（暗号化）。 |
| **サブ** `affirmations/{affirmationId}/coach_share_rounds/{roundId}` | A-11: クライアント「コーチへ送信」1 回につき 1 ドキュメント。 |
| **サブ** `.../coach_share_rounds/{roundId}/coach_comment_versions/{versionId}` | A-11: コーチコメントの版（追記のみ）。 |

- **下書き（穴埋め・未発行・テンプレ入力中）の永続化**: **`users/{uid}/affirmation_drafts/{profileId}`** に暗号化スロット JSON（現行実装）。作成モーダルで **[保存]** した内容がここに載る。**まだ `affirmations` に親ドキュメントが無い段階**でもよい（発行時に親＋`published/current` を作成）。
- **履歴**: 上表のとおり `history/{historyId}`。
- **削除**: **物理削除**。親削除時に **`published`・`history` をバッチ等でまとめて削除**する。復元不可である旨を確認ダイアログで必須表示する。
- **紐づけの切り離し**: 「発行」した時点で編集対象は **文書の `body` のみ**（`published/current`）。slotId 単位の ID は完成文書には持たない。

#### 3.3.1 編集中のデータはどこに保存されるか

「編集中」は **種類が 2 つ**あります。

| 状況 | Firestore に保存されるか | 保存場所 |
|------|---------------------------|----------|
| **① 未発行の穴埋め（作成モーダル）** | **[保存]** したときだけ永続化 | **`affirmation_drafts/{profileId}`**（暗号化スロット）。モーダル内の未保存の変更は **ブラウザのメモリ（React state）** のみ。 |
| **② 発行済み本文の編集（編集モーダル）** | **[保存]** するまで | **Firestore には書かない**。編集内容は **メモリ上のみ**。[保存] で **`published/current`** を更新（＋任意で `history` 追加）。 |

- **閉じる**（作成モーダル）: 設計どおり **完全破棄**（開く前のスロットへ復元）— セッション中未 [保存] の編集は捨てる。既に [保存] 済みの下書きは Firestore に残る。
- **複数の「入力中」**を同じプロファイルで同時に持つ必要が出た場合は、`affirmation_drafts` のキーを `profileId` 単体から **`affirmationId` 連動**に拡張するなど、別途検討（現状は 1 プロファイル 1 下書き想定）。

### 3.4 作成モーダルの「閉じる」挙動（決定・旧: キャンセル）

- **決定**: 作成時の **閉じる**（フッターボタン・**右上×**・オーバーレイクリック・Esc 等、モーダルを閉じる操作）は **完全破棄**（未保存入力は保存しない。開いた直前のスロットに戻す）。
- **UI 文言**: ユーザー向けは **「閉じる」** とし、「キャンセル」は使わない（誤解を減らす）。
- モーダル表示中は、他アプリにフォーカスが移っても、モーダルが閉じるまでは入力状態を保持する前提とする。
- 破棄のタイミングは **閉じる** 押下時（または同等の破棄操作時）とする。
- `保存` を押した内容のみ、下書き（draft）として残す。

### 3.5 UI サブメニュー（選択・作成・編集・履歴）— 要件サマリ

| メニュー | 要点 |
|----------|------|
| **選択** | 「入力中」「発行済」を一覧表示し、[表示]で内容表示。一覧にファイルが無い場合はメニュー非活性。**削除**は物理削除・確認ダイアログ必須（履歴も消える旨）。**名称変更**可。同一表示名は不可（UI チェック＋エラー）。一覧の表示項目（最小セット）: **名称、状態、最終更新日、プロファイル名**。 |
| **作成** | モーダル表示。穴埋めテンプレートで入力。[保存]＝下書き、[発行]＝完成文書として登録（名称付与・デフォルト名あり）、**[閉じる]**（フッター）および **右上×**＝モーダルを閉じ、未保存の入力は破棄（開く前のスロットに戻す）。 |
| **編集** | 入力中はテンプレ継続編集。発行済は **本文全体を編集モーダル**（**Markdown 直接編集**。スロットへの逆変換はしない。決定: [04_TRIAL_28_IMPLEMENTATION_DECISIONS.md](./04_TRIAL_28_IMPLEMENTATION_DECISIONS.md) §9.7 #1）。[保存]（**本文**）時のみ「前バージョンを履歴に残すか」確認→残す／上書きのみ（§9.7 #2）。履歴スナップショットには **当時の名称も暗号化して含める**（§9.7 #2b）。名称変更のみでは当該ダイアログは出さない。 |
| **履歴** | **発行済みのみ**（A-9）。一覧は **各版の日時（`savedAt`）** と **履歴件数（`history` ドキュメント数＝「履歴を残す」で積んだ版の数）** を先行表示項目とする（[04_TRIAL_28_IMPLEMENTATION_DECISIONS.md](./04_TRIAL_28_IMPLEMENTATION_DECISIONS.md) §9.7 #5b・§9.7.3）。版を選びモーダルで閲覧。[戻る]で閉じる。**参照のみ**（復元は後続可）。 |

#### 3.5.1 作成モーダル（実装状況・2026-03）

- **UI**: `/trial_4w` のアファメーションタブで、[作成] を押すと **全画面オーバーレイ**（`position: fixed`）＋前面パネルで他操作をブロックする。実装: `src/components/trial/TrialAffirmation.tsx`（`createPortal(..., document.body)`）。
- **ヘッダー右上**: **×（閉じる）** ボタン。フッターの **[閉じる]** と同じ動作（スロット復元・モーダル終了）。
- **フッター**: 最下段に **[保存]** **[発行]** **[閉じる]**（旧称キャンセル相当。ユーザー向け文言は「閉じる」）。本文・プレビューはモーダル内スクロール。
- **[保存]**: 現プロファイルのスロット状態を暗号化し、`users/{uid}/affirmation_drafts/{profileId}` に保存（`saveAffirmationDraft`）。成功後は **閉じる時の復元基準**を現在の入力に更新する。
- **[発行]**: プレビューと同じ生成結果を 1 本文として `encrypt` し、案 B のパスに書き込む。実装: `src/lib/firestore.ts` の **`publishAffirmation`**（親 `users/{uid}/affirmations/{affirmationId}` にメタ、`published/current` に `encryptedBody`）。発行名はモーダル内入力。空欄時は **`新規アファメーション（YYYY-MM-DD）`** 形式のデフォルト名。**成功後**は **`deleteAffirmationDraft`** で `affirmation_drafts/{profileId}` を削除し、プレビュー用スロットもクリア。サブメニュー表記は **「作成」** に戻る（「新規」と誤解されないよう、途中から再開した場合も **下書きあり＝「作成中」** とする）。
- **サブメニュー「作成」／「作成中」**: 初回や下書きが無い状態では **「作成」**。次のいずれかで **「作成中」** に切り替える — (1) 画面ロード時に `affirmation_drafts/{profileId}` が既にある、(2) モーダル内で **[保存]** が一度でも成功した。発行成功で下書き削除後は再び **「作成」**。
- **下書きのみ手動で消す場合**: 現段階は **Firebase コンソール**で `affirmation_drafts` の該当ドキュメントを削除するか、将来の **「下書きを破棄」** UI を想定。発行フローではアプリ側で自動削除する。
- **[閉じる]**（フッター） / **右上×** / オーバーレイクリック / **Esc**: モーダルを閉じ、**モーダルを開いた直前のスロット状態**に戻す（未 [保存] の編集は破棄。[保存] 済みなら直前の保存内容相当に戻る）。
- **選択タブでの発行済み一覧**: **A-7 実装済**（`listUserAffirmations`・表示・行内名称変更・`deleteAffirmationFully`）。詳細は [04_TRIAL_28_IMPLEMENTATION_DECISIONS.md](./04_TRIAL_28_IMPLEMENTATION_DECISIONS.md) §9.6。

### 3.6 初期表示（トライアル画面）

- 初期は **プレビューのみ**（作成フォームは非表示）。
- 前回選択中のアファメーションがあればその内容を表示。
- 未作成なら新規作成相当の導線（作成メニューでモーダル）。

**UI 状態の永続化（決定）**: **localStorage は使わず**、`users/{uid}` に **`trialAffirmationMeta`**（map）を保存する。

| フィールド | 型 | 説明 |
|------------|-----|------|
| `lastSubmenu` | `null` \| `'select'` \| `'create'` \| `'edit'` \| `'history'` | 前回開いていたサブメニュー。**null** は「プレビューのみ」表示。 |
| `lastSelectedAffirmationId` | `null` \| string | 一覧で最後に選んだ `affirmations/{affirmationId}` の ID（A-7 以降で更新）。 |

- 実装: `src/lib/firestore.ts` の `updateTrialAffirmationUiMetaFields`、`src/types/auth.ts` の `TrialAffirmationUiMeta`。

### 3.7 ロール別権限（確定）

| ロール | アファメーション |
|--------|------------------|
| **クライアント（owner）** | 自データに対する **読取・作成・編集・削除・発行・履歴** の全操作。**A-11**: 条件付きで「コーチへ送信」、コーチからのコメントは **読取のみ**（最新版および履歴の見せ方は UI で定義）。 |
| **コーチ** | **発行済みかつ共有 OK** の本文（`published/current`）を **閲覧**。**下書き・本文の編集・削除・編集履歴（`history`）は行わない**。**A-11**: 担当は **`coach_client_assignments`** で判定。`coach_share_rounds` に対応する **`coach_comment_versions` への追記**（版を増やす）を行う。コーチモード時の画面遷移: 担当クライアント一覧 → 選択 → 共有対象アファメーション表示 → コメント。詳細は [03_A11_COACH_SHARING_SCHEMA_DRAFT.md](./03_A11_COACH_SHARING_SCHEMA_DRAFT.md)。 |
| **管理者** | **IF 上からユーザーコンテンツ（アファメーション本文）は閲覧しない**（既存方針と整合）。 |

**アクティブテーマ**: クライアントの **`users/{uid}.activeCoachingAffirmationId`** が「いま実施中の 1 テーマ」。複数の発行済み文書があっても、コーチングの主眼はこの ID に合わせる想定（過去文書へのコメントはデータ上は別 `affirmationId` で保持可能）。

### 3.8 同意ゲートとの関係

- `/trial_4w` 前に同意が必須のため、アファメーション機能は **同意済みユーザ** を前提とする（追加の依存なし）。

### 3.9 現状実装での確認項目（画面テスト用）

※ ここでは「いま入っている実装」を前提に、画面を見ながら段階検証できる観点を列挙する（将来 A-7〜A-9 の本実装や、A-11 の Firestore 読み取りが入るとこの項目は更新される）。

#### 0. 共通（表示モード）

- ログインしているユーザで、ヘッダー右上メニューの「表示モード」を `コーチ` に切り替えられる（表示可能ロールは `coach` / `senior_coach` 想定）
- `/trial_4w` のタブ `アファメーション` を開くと、`TrialAffirmation` が表示される

#### 1. クライアント（通常表示 / `client` モード）

- 初期表示は「プレビューのみ」になり、サブメニュー `選択` は **押下可能**（一覧が空でも開き、案内が出る）
- 下書きが無いときはサブメニューが **`作成`**、下書きありまたは [保存] 成功後は **`作成中`** と表示される
- `選択` を押すと一覧表（選択列・名・作成状態・最終更新・プロファイル名）と、フッターの **表示／名称変更／削除／閉じる** が表示される（行クリックは**一覧上の選択**のみ。下書き行を含み最終更新降順）
- `作成` / `作成中` は押下可能になっている
- `編集` と `履歴` は、選択済み ID が無いため非活性になっている
- `作成` を押すと **モーダル** が開き、穴埋めテンプレート・プレビュー・**右上×**・フッター **[保存][発行][閉じる]** が表示される（背面はオーバーレイでブロック）
- `保存` を押すと、モーダル内に「最終保存: …」が表示される（下書きは暗号化して `users/{uid}/affirmation_drafts/{profileId}` に保存）
- `発行` を押すと、Firestore に `users/{uid}/affirmations/{affirmationId}` と `published/current` が作成され、**下書きドキュメントが削除**され、成功メッセージが出てモーダルが閉じる。サブメニューは再び **`作成`** に戻る（`trialAffirmationMeta.lastSelectedAffirmationId` が更新される）
- `閉じる`・**右上×**・オーバーレイクリック・Esc でモーダルが閉じ、**開く前のスロット状態**に戻る
- 「← プレビューのみ表示に戻る」を押すと、プレビューのみへ戻り、サブメニュー状態が `trialAffirmationMeta` に保存される（`作成` 中は閉じると同様にスロットを復元）

#### 2. コーチ閲覧（`coach` モード / A-11 UI骨格の状態確認）

- コーチモードのとき、プレビュー領域の見出しが `コーチ閲覧（導線）` になる
- コーチモードのとき、サブメニューは `選択` のみ押下可能で、`作成` / `編集` / `履歴` は非活性になる
- `選択` を押すと、プレースホルダーとして `共有閲覧（選択）` が表示される（現状は担当クライアント一覧・`sharedWithCoach` フィルタ・`published` の Firestore 読み取りは未実装のため）
- コーチモードでは「戻る（プレビューのみ）」ボタンが基本的に表示されない（骨格仕様）

#### 3. Firestore で期待する通信（失敗時の切り分け）

- `/trial_4w` の `アファメーション` を開いた時点で、下書き取得 `getAffirmationDraft` と `getUserProfile` が呼ばれる（現状の UI 骨格は Firestore 読み取りを止めていない）
- 作成モーダル内の `保存` は、下書き保存（暗号化）と、暗号文更新が発生する
- 作成モーダル内の `発行` は、`publishAffirmation` により親ドキュメント＋`published/current` の書き込みに加え、成功時に `deleteAffirmationDraft` で下書き削除が発生する
- サブメニュー操作（`選択` / `← プレビューのみ表示に戻る` 等）では、`users/{uid}.trialAffirmationMeta` のドット更新が発生する

## 5. 入力データの暗号化

- 要件定義（7.3）およびプライバシーポリシーに基づき、**Firestore に保存するクライアント・コーチの入力データは暗号化**する。
- **対象**: アファメーションの穴埋め入力データ、完成文（body）、履歴、および 28日間トライアルの日次記録・コーチのコメント等。**一覧表示用の `lastPreviewText`（直近プレビュー文字列）も内容データのため暗号化対象**とする。
- **方針**: 既存の `src/utils/encryption.ts`（ユーザー uid ベースの鍵など）を流用し、保存前に暗号化・読み込み時に復号する。28日間トライアル全体の暗号化方針は [04_TRIAL_28_IMPLEMENTATION_DECISIONS.md](./04_TRIAL_28_IMPLEMENTATION_DECISIONS.md) セクション 10 を参照。
