### 1.4 決定: トライアル開始フロー（ランディング + 同意ゲート）

- **ホーム（未ログイン）**: ボタン「**試してみる**」→ `GET /trial_4w/landing`
- **ランディング（/trial_4w/landing）**:
  - 2/2 の **AIコーチ** の「やってみる」のみ有効
  - **未ログイン**なら `GET /login?next=/trial_4w`（ログイン必須）
  - **ログイン済み**なら `GET /trial_4w`
- **ログイン後**: `GET /post-login?next=/trial_4w` に遷移して自動分岐
  - **同意済み**なら `next` へ
  - **未同意**なら `GET /consent?next=/trial_4w`（同意必須）
- **同意ページ（/consent）**:
  - `users/{uid}.consents` に同意情報を保存（termsVersion/privacyVersion は **日付**）
  - 保存後 `next` へ遷移

（実装: `src/app/trial_4w/landing/page.tsx`, `src/app/login/page.tsx`, `src/app/post-login/page.tsx`, `src/app/consent/page.tsx`, `src/lib/consent.ts`, `src/lib/firestore.ts:updateUserConsents`）

## 9. アファメーション要件（追加）

### 9.1 アファメーションのプロファイル（穴埋めテンプレート）

| ----------- | ------------------------------------------------------------------------------------- |
| **要件**      | アファメーション入力は **穴埋め式**。穴の並び・前後の文字列は **プロファイル** で定義。将来的にユーザーがプロファイルの作成・編集・選択ができる構造にする。   |

### 9.2 入力データの ID 付与と紐づけ


| 項目      | 内容                                                                                                                                                   |
| ------- | ---------------------------------------------------------------------------------------------------------------------------------------------------- |
| **要件**  | プロファイルと、そのプロファイルから生成された **入力データ** を **ID で紐づける**。穴の値は **別プログラム（目標設定・行動策定などのステップ）** で作成し、最後にまとめる運用に対応する。                                              |
| **紐づけ** | 各入力データに ID を付与。プロファイル側の各「穴」は **slotId** で一意。入力データは「どのプロファイルのどの slotId に対応するか」を参照で持つ。複数ステップの結果を 1 つのアファメーションにまとめる際に、slotId 単位でどの入力データを埋めるか指定できるようにする。 |


### 9.3 作成したアファメーションの編集と履歴


| 項目     | 内容                                                                                                                                                                                 |
| ------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **要件** | 穴埋めで組み立てたあと、**全体** のニュアンス・つなぎ・内容を修正できるようにする。**完成した文を書式付きテキストとして出力** し、それをユーザーが利用するアファメーションとする。この段階では個別センテンスの ID 紐づけはなくし、**1 つの文書** として扱う。文書には **タイトル・ID** を付与。**バージョンアップの履歴** を残す。 |
| **流れ** | 穴埋めで組み立て → 「発行」で **Markdown 本文** に変換（テンプレは `[[slotId:最大文字数]]` 記法）→ ユーザーが完成文を直接編集 → **発行済み**の保存時のみ「履歴を残す／上書き」を選択。履歴は **発行済みのみ** 対象。                                                 |

### 9.7.2 Firestore の「親 `updatedAt`」と `encryptedLastPreviewText`（用語の説明）

#### 親ドキュメントの `updatedAt` を本文保存で「触る」とは？

Firestore では **2 段所**に日時があります。

| 場所 | ドキュメント | いま「最終更新」に使っている例 |
|------|----------------|-------------------------------|
| **親** | `users/{uid}/affirmations/{affirmationId}` | **`listUserAffirmations`** が読む **`updatedAt`**（選択タブ一覧の並び・「最終更新」列） |
| **本文** | `.../affirmations/{id}/published/current` | 本文の `updatedAt`（発行・本文更新のたびに更新する想定） |

**「親の `updatedAt` を触る」**＝**本文を保存したときに、親ドキュメントにも `updateDoc` で `updatedAt: serverTimestamp()` を書く**という意味です。

**なぜ必要か**

- 一覧は **親だけ**を読んでいるため、**`published/current` だけ更新**して親の `updatedAt` を古いままにすると、「本文を編集したのに **選択一覧の最終更新が変わらない**」「**並び順が実態とずれる**」ことが起きます。
- **名称変更**の `updateAffirmationTitle` では既に親の `updatedAt` を更新しているので、**本文保存でも同様に親を更新**すると挙動が揃います。

**決定（#3d）**: 本文保存（履歴あり／なしどちらでも）の成功時に、**親の `updatedAt` を必ず更新する**。

---

#### `encryptedLastPreviewText`（親の任意フィールド）とは？

- **目的のイメージ**: 一覧表で「本文の先頭数十文字のプレビュー」を出したいとき、**全文を読まず・毎回全文復号しない**ようにするため、**親ドキュメントに「短い暗号化スニペット」だけ持つ**という設計案（[04_AFFIRMATION_DESIGN.md](./04_AFFIRMATION_DESIGN.md) の親メタ例）。
- **現状**: 選択一覧は **タイトル・状態・日時など**で足りており、**このフィールドは未使用**。
- **決定（#3e）**: **A-8 では書き込まない（後回し）**。将来、一覧にプレビュー列を足すときに検討する。

### 9.7.3 A-9 履歴タブ：表示項目と「改定回数」（#5b 補足）

- **日時**: 各履歴行は **`savedAt`**（その版を履歴に残した日時）を表示する。並び替えは **降順（新しい版が上）** が分かりやすい。
- **改定回数と ID の関係**:
  - Firestore の **`history/{historyId}`** の `historyId` は **`addDoc` の自動 ID**（ランダム）なので、**「第 3 改定」のような連番ではない**。
  - いわゆる **「何回分の履歴があるか」** は **`history` 配下のドキュメント数を数える**とよい（＝ユーザー質問の **「ID の数」＝ドキュメント件数** で一致）。
  - UI では例えば **「履歴：全 5 件」**のように **件数**を出す／または新しい順に **1 からの通し番号**を表示用に付与してもよい（DB に連番フィールドを増やす必要はない）。

**本文プレビュー・当時の名称**: スキーマには **`encryptedTitle` / `encryptedBody`** があるため、A-9 で詳細モーダルを開いたときに **復号して表示**可能（一覧の最小項目は #5b の **日時＋件数** で開始し、必要なら行に **タイトル** を足す）。

### 9.7.4 「共有スナップショット」とは何か（#6c・用語）

**いまの想定（採用している考え方）**

- クライアントが **共有 ON** にすると、コーチは **`published/current` の最新内容**を（ルール整備後）読める。
- クライアントがあとから本文を編集すると、**コーチが次に見えるのも編集後の最新版**（同じドキュメントが更新されるため）。

**「共有スナップショット」という別案（※現時点では採用しない想定の例）**

- **共有した瞬間の本文**を、別コピーとして固定保存し、**クライアントが後から `published/current` を書き換えても、コーチには共有当時の文面だけ**を見せ続める方式。
- 使われ方の例: 「この文面を見たうえでコーチがコメントした」という **文脈を固定**したい、監査・同意の範囲を **当時の表示に合わせたい**、など。
- **現設計は「常に最新を共有に反映」**に近いため、**A-8 実装の前提としては不要**。将来、プロダクト・法務で必要になったときに **別仕様**として検討する。

## 11. 実装の進め方（推奨順）

1. **認証・アクセス（1-x）** と **サブスクの器（2-x のうち 2-1, 2-2）** を決め、トライアル画面の「入れる/入れない」と「残り日数」の有無を確定する。
2. **データ保存先・スキーマ（3-1, 3-2）** を決め、Firestore のパスと「1日分」のフィールド一覧を書く。
3. **アファメーション（9-x）**: 指定プロファイル 1 本の構造を固定し、穴埋め入力・入力データの ID・プロファイルとの紐づけをスキーマに反映する。完成文の「1 文書 + 履歴」をどこまで今回実装するか決める（[04_AFFIRMATION_DESIGN.md](./04_AFFIRMATION_DESIGN.md) 参照）。
4. **各タブの入力項目（4-x）** を、プロトタイプと要件の記録観点で揃え、スキーマと対応表を作る。
5. **入力データの暗号化（10）**: 保存前に暗号化・読み込み時に復号する処理を組み込む。既存 `src/utils/encryption.ts` を拡張または流用する。
6. **セキュリティルール（6-1）** を追加し、本人のみ read/write をデプロイする。
7. アファメーション表示・穴埋め入力・朝・晩の入力・保存・読み込みを実装し、週・月は「集約表示」からでよいか決める（3-4, 4-4, 4-5）。
8. コーチ共有（5-x）は、必要ならデータにフラグだけ入れておき、コーチ側画面は後フェーズにする。コーチの入力（コメント等）も暗号化対象とする。
