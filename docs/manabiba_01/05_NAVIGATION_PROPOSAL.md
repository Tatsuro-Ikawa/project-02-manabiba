# グローバルナビゲーションとタブ（アプリ）ナビゲーションの整合性提案

## 1. 目的

モバイル・タブレットで「ハンバーガーメニュー」と「左ツールバーを開くボタン」が並び、役割が重複・混同しやすい状態を整理するため、**グローバルナビゲーション**と**タブ（アプリ）ナビゲーション**の役割定義と、小さい画面での見せ方を検討する。

---

## 2. 現状の整理

### 2.1 二種類のナビゲーション

| 種類 | 役割 | 想定項目（例） | 主な利用場面 |
|------|------|----------------|--------------|
| **グローバルナビゲーション** | サイト全体の情報・制度・問い合わせへの導線 | サービス、料金プラン、利用規約、プライバシーポリシー、FAQ、お問合せ、ランディング | ログイン前後を問わず「サイトについて知りたい」「問い合わせたい」とき |
| **タブ（アプリ）ナビゲーション** | アプリ内の作業フロー・機能への導線 | ホーム、スタート、ゴール、計画、実績、反省、マイページ | ログイン後、「PDCAを回す」「目標を見る」など利用時 |

- **グローバル** ＝ 「サイトとしてのこころ道場」の情報（マーケティング・法務・サポート）。
- **タブ（アプリ）** ＝ 「利用者としてやること」の順路（ホーム→スタート→ゴール→…→マイページ）。

### 2.2 現在のプロトタイプでの表現

- **ハンバーガーメニュー** → グローバルナビ（サービス、料金、FAQ、お問合せ、ランディング等）。
- **左ツールバー（☰で開く）** → タブ（アプリ）ナビ（ホーム、スタート、ゴール、計画、実績、反省、マイページ）。

問題点: モバイル・タブレットで「☰」と「≡」のような似たボタンが並び、どちらが「サイトの情報」でどちらが「アプリのメニュー」か分かりにくい。

### 2.3 01_UI_UX_DESIGN.md との関係

- **4.3.1 グローバルナビゲーション**では、項目として「ホーム、自己理解、テーマ選択、目標設定、PDCA、振り返り、コーチングプログラム」が挙がっている。
- これは**アプリのワークフロー**に近く、現プロトタイプの「左ツールバー」の内容に対応する。
- 一方、プロトタイプのハンバーガーは「サービス、料金、FAQ、お問合せ」など**サイト情報**を担当している。

→ 用語の対応をはっきりさせるとよい。

- **グローバルナビゲーション**  
  - 案A: 「サイト情報」に限定（サービス、料金、利用規約、FAQ、お問合せ 等）  
  - 案B: 上記に加え「アプリの入口」も含む（ホーム／マイページへのリンク等）
- **タブナビゲーション（アプリナビ）**  
  - ホーム、スタート、ゴール、計画、実績、反省、マイページ（＝UI_UX_DESIGNの「グローバル」に書かれている項目と実質同じ役割）

設計書では「グローバル＝ヘッダー／ハンバーガーで出すもの」を、**サイト情報**と**アプリメニュー**のどちら（または両方）と定義するかを決めると、一貫した設計にしやすい。

## 3. 考え方の軸

### 3.1 軸1: 入口の数（ひとつにするか、ふたつに分けるか）

- **ひとつにまとめる**  
  - モバイル・タブレットでは「メニュー」ボタンは1つだけ。  
  - 開いたパネル／ドロワー内で「サイト情報」と「アプリメニュー」を**セクション分け**して表示する。  
  - メリット: 迷いにくい。「メニューを開けば全部ある」と理解しやすい。  
  - デメリット: 項目数が増えると長くなる。見出し・区切り線で明確に分ける必要がある。

- **ふたつに分ける**  
  - 「サイト情報」用と「アプリメニュー」用で、**アイコン・ラベルを変えた2ボタン**にする。  
  - 例: [ℹ️ サイト] [≡ メニュー] や [サイト情報] [ホーム・スタート等]。  
  - メリット: 役割が明確。「料金だけ見たい」ユーザーはサイト側だけ開けばよい。  
  - デメリット: ヘッダーが狭いと2つ並べると窮屈。アイコンとラベルで「サイト」と「アプリ」の違いをはっきり伝える必要がある。

### 3.2 軸2: 階層でどう扱うか（アプリを主にするか、サイトを主にするか）

- **アプリを主にする**  
  - メニューの**上段**にアプリナビ（ホーム、スタート、ゴール、…、マイページ）、**下段**に「サイトについて」として サービス・料金・FAQ・お問合せ をまとめる。  
  - ログイン後の利用を想定した設計。  
  - 「まずアプリで作業し、必要ならサイト情報へ」という流れに合う。

- **サイトを主にする**  
  - 逆に、上段が「サービス・料金・お問合せ」、下段が「ホーム・スタート・…・マイページ」。  
  - ログイン前や、まず「こころ道場とは」「料金」を見たいユーザー向け。

- **文脈で切り替える**  
  - ログイン前: ひとつのメニューに「サイト情報」を中心にし、アプリ入口（ログイン／ランディング）を少しだけ出す。  
  - ログイン後: 同じ1メニューでも「アプリメニュー」を上に、「サイト情報」を下に配置する。  
  - 同じUIパターンで、中身の順序だけ変えると理解しやすい。

### 3.3 軸3: デスクトップと小さい画面の対応関係

- **デスクトップ**  
  - 左ツールバー ＝ アプリナビ（常時表示）。  
  - ヘッダー ＝ ロゴ・タイトル ＋ **グローバル（サイト情報）** のリンク数個 ＋ ログイン／アバター。  
  - グローバルは「サービス」「料金」「FAQ」「お問合せ」などテキストリンクでよい。

- **タブレット・モバイル**  
  - 左ツールバーは収納（スライドアウト）。  
  - ヘッダーには「サイト情報」と「アプリメニュー」の**どちらをどう出すか**を、上記の「1つにまとめる／2つに分ける」「アプリ主／サイト主」に合わせて決める。

一貫性の目安:  
「デスクトップで左に出ているもの ＝ 小さい画面では『アプリメニュー』として開くもの」「ヘッダーにあるサイトリンク ＝ 小さい画面では『サイト情報』として開くもの」と対応させると分かりやすい。

## 4. 選択肢の整理（モバイル・タブレット）

### 案A: ひとつのメニューに統合（セクション分け）

- ハンバーガー（または「メニュー」）は**1つだけ**。  
- 開いたパネル内で:  
  - **上**: アプリナビ（ホーム、スタート、ゴール、計画、実績、反省、マイページ）  
  - **区切り線**  
  - **下**: サイト情報（サービス、料金プラン、利用規約、プライバシーポリシー、FAQ、お問合せ）、必要ならランディング  
- 左ツールバー用の「☰」は廃止。  
- メリット: ボタンが1つで、ハンバーガーとツールバー開きの重複が解消される。  
- デメリット: 項目数が多くなるので、見出し（「アプリ」「サイトについて」など）と余白で区切りをはっきりつける必要がある。

### 案B: ふたつのボタンで役割を明確に分ける

- **ボタン1**  
  - ラベル例: 「サイト」「情報」「サイト情報」、アイコン例: ℹ️ や 本のマーク  
  - 開く内容: サービス、料金、利用規約、プライバシー、FAQ、お問合せ  
- **ボタン2**  
  - ラベル例: 「メニュー」「ホーム等」「アプリ」、アイコン例: ≡ や 縦三点  
  - 開く内容: ホーム、スタート、ゴール、計画、実績、反省、マイページ（＝左ツールバーと同じ）  
- メリット: 「サイトのことを知りたい」と「アプリの画面を切り替えたい」が明確に分かれる。  
- デメリット: ヘッダーに2つ並ぶため、アイコン＋短いラベルで違いが一目で分かるようにする必要がある。

### 案C: アプリメニューを主にし、サイト情報はその中の一項目にする

- メニューは**1つ**。  
- 中身は**アプリナビを上に**（ホーム、スタート、ゴール、計画、実績、反省、マイページ）。  
- その下に「**サイトについて**」という1項目を置き、タップで展開または子リスト表示（サービス、料金、FAQ、お問合せ）。  
- メリット: ログイン後の主用途（アプリの移動）が前面に出る。サイト情報は「必要な人だけたどる」構造にできる。  
- デメリット: サイト情報が一段深くなるので、「料金だけすぐ見たい」という動線はやや長くなる。

### 案D: 文脈で中身を変える（ログイン前はサイト主、ログイン後はアプリ主）

- **ログイン前**  
  - メニュー1つ。中身は「サイト情報」中心（サービス、料金、FAQ、お問合せ、ランディング、ログイン）。  
- **ログイン後**  
  - 同じ1メニューでも、中身を「アプリナビ」中心に切り替え（ホーム、スタート、…、マイページ）、下に「サイトについて」をまとめる。  
- メリット: 今いる状態（未ログイン／ログイン後）に合わせて、必要な方が上に来る。  
- デメリット: 実装とテストのパターンが増える。遷移時（ログイン直後など）にメニューの中身が変わることをユーザーが理解できるかは、ラベルや説明で補う必要がある。

## 5. 推奨の方向性（考え方として）

1. **用語を設計書でそろえる**  
   - **グローバルナビゲーション** ＝ サイト情報（サービス、料金、利用規約、FAQ、お問合せ 等）を指すと定義する。  
   - **タブナビゲーション／アプリナビゲーション** ＝ ホーム、スタート、ゴール、計画、実績、反省、マイページ と定義する。  
   - 01_UI_UX_DESIGN.md の「4.3.1 グローバルナビゲーション」の項目を、上記のどちらに合わせるか（または「両方含む」と明記するか）を決める。

2. **モバイル・タブレットでは「メニューを1つにまとめる」を優先候補とする**  
   - ハンバーガーと左ツールバー用ボタンの**重複をなくす**ため、**案A（1メニューに統合、セクション分け）** を第一候補とする。  
   - そのうえで、項目数や利用データを見て「サイトだけ開く人が多い」などが分かれば、**案B（2ボタンで役割分離）** も検討する。

3. **1メニューにする場合の並び**  
   - ログイン後を主に考えるなら「アプリナビ（上） → 区切り → サイト情報（下）」とする。  
   - ログイン前は「サイト情報＋ランディング・ログイン」を上にしてもよい（案Dの簡易版）。

4. **デスクトップとの対応**  
   - 左ツールバー ＝ アプリナビ（現状どおり）。  
   - ヘッダー右側のリンク ＝ グローバル（サイト情報）。  
   - 小さい画面の「1メニュー」は、この2つを**1つのドロワーにまとめたもの**と説明できるようにする。
