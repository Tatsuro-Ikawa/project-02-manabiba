# 実装手順：ホーム画面とこころのトライアル画面

## 📋 目的

プロトタイプで確定した「ホーム画面」と「28日間 こころのトライアル」4画面を、Next.js 上で実装し、外（本番に近い環境や実機）で確認できるようにするための手順です。

**補足（2026-04〜）**: トライアル **Vertex AI** のセットアップ・REST 仕様・トラブルシュートは [../VERTEX_AI_TRIAL_IMPROVEMENT.md](../VERTEX_AI_TRIAL_IMPROVEMENT.md) に集約した。朝・晩は `/api/ai/improvement`、週タブは `/api/ai/weekly-report`（Aiレポート）と `/api/ai/weekly-improvement`（Ai改善提案・詳細表示時）。

---

## 実装レベル（今の段階でやること・やらないこと）

**今の段階では、DB 連携・認証システムは実装しない。プロトタイプと同様、画面イメージ重視のレベルで進める。**

| やること | やらないこと |
|----------|----------------|
| ルーティング（`/`、`/mypage`、`/trial_4w`） | Firestore 等の DB 連携 |
| レイアウト・スタイル（サイドバー、1026px、デザインシステム） | Firebase Authentication 等の認証実装 |
| 画面構成（ホーム、トライアルのタブ切替） | ログイン状態の永続化・ガード |
| 表示の切替（ログイン前／後風）は **モックで模擬**（URL パラメータやローカル state 等） | データの保存・読み込み（実データ） |
| 外で確認できるようにビルド・デプロイ | 本番用のセキュリティ・エラーハンドリング |

- **イメージ**: HTML プロトタイプを Next.js のコンポーネントとルートで再現し、「見た目と遷移」を確認できる状態にする。認証・DB は後段階で組み込む。

## 【1】ルーティングと画面構成の設計

**目的**: どのURLでどの画面を表示するか、既存の `/` や `/mypage` とどう役割分担するかを決める。

### 1.1 ルーティング設計（確定）

基本は A 案とし、以下で実施する。

#### 1.1.1 ホーム・ランディング・マイページ

| パス | 画面 | 備考 |
|------|------|------|
| **`/`** | **ホームページ** | 未ログイン／ログイン済み**風**の表示切替。今の段階ではモック（例: `?logged_in=true`）で切替。将来は認証状態で判定。プロトタイプ SCREEN-000 相当。 |
| （別途） | **ランディングページ** | 別途作成。URL は後で決定（例: `/landing` 等）。 |
| **`/mypage`** | **マイページ** | 現状のまま残す。今後、個人リソースのダッシュボード風画面への変更を検討する。 |

- トライアルのサイドバー「ホーム」は **`/`** へリンクする。

#### 1.1.2 28日間トライアル（`/trial_4w`）— **案2で実施**

- **ベースパス**: **`/trial_4w`**（この1パスのみ）
- 「アファメーション」「朝・晩」「週」「月」の4コンテンツは、**同一パス内でクライアント側のタブ切り替え**で表示する。
- **共通部分**: ヘッダー、左サイドバー、メニューバー（アファメーション／朝・晩／週／月）は共通のため、**1つのレイアウト**でラップし、**メインエリアだけを state でコンポーネント切り替え**する。
- **実装の考え方（案2）**:
  - **URL**: `/trial_4w` のみ。必要に応じて `?tab=affirmation` や `?tab=morning-evening` などクエリでタブを保持し、ブックマーク・共有時に同じタブを開けるようにする。
  - **表示切替**: メニューバーのクリックで、表示するコンポーネント（アファメーション／朝・晩／週／月）を state（またはクエリと同期）で切替。サブルートは作らない。
  - **構成**: `app/trial_4w/page.tsx` が1ページで、共通レイアウト（ヘッダー・サイドバー・メニューバー）と、現在のタブに応じたメインコンテンツを描画する。

#### 1.1.3 今後の拡張（商品ごとのルート）

- **28日間トライアル以外**に、7日間プログラムやその他テーマ別商品を追加する想定。
- **商品ごとにルートを分ける**と分かりやすい（例: `/trial_4w`、`/program_7d`、その他テーマ用）。
- 今回の `/trial_4w` を「1商品＝1ベースパス」のパターンとしておき、他商品も同様に配下でサブ画面を持つ形にすると拡張しやすい。

### 1.2 ルーティング一覧（今回のスコープ）

| URL | 画面 |
|-----|------|
| `/` | ホームページ |
| `/mypage` | マイページ（将来ダッシュボード風に変更検討） |
| `/trial_4w` | 28日間トライアル。メニューバーで「アファメーション／朝・晩／週／月」をクライアント側で切替。必要なら `?tab=affirmation` 等のクエリでタブを保持。 |

### 1.3 補足（実装方針）

- **案2**: `/trial_4w` は**1ページ**。共通レイアウト（ヘッダー・サイドバー・メニューバー）を描画し、メニューバーの選択に応じてメインエリアのコンポーネント（アファメーション／朝・晩／週／月）を state で切り替える。クエリ `?tab=...` と同期させると、URL でタブを復元・共有しやすい。
- 商品ごとにルートを持つ方針は、7日間プログラム等を追加する際も `/program_7d` のように並列で増やしやすく、おすすめ。

## 【2】共通レイアウト・デザイントークンの準備

**目的**: プロトタイプの見た目と仕様（1026px、色・余白）を実装側で再現しやすくする。

### 2.1 やること

- **デザイントークン**
  - `docs/manabiba_01/02_DESIGN_SYSTEM.md` およびプロトタイプの `prototype-common.css`（`--color-primary` など）を参照し、Tailwind の theme 拡張や CSS 変数で同じ値を定義する。
- **アイコン**
  - 実装版では **Google Fonts の Material Symbols**（無料）を使用する。読み込み方法・アイコン名は DESIGN_SYSTEM の「5.1.1 実装版の標準：Google Fonts 無料アイコン（Material Symbols）」およびプロトタイプ絵文字との対応表を参照する。
- **共通レイアウト**
  - **ホーム**: 左サイドバー（80px）＋ main-content（最大 1300px）＋ content-grid（左 1fr / 右 328px）。左カラム（content-left）の**最大幅 948px**（`--content-left-max-width`）。右カラム（content-right）は 328px（いちおしサイト表示用）。
  - **トライアル**: 左サイドバー（80px）＋ trial-main（**最大 948px**、同一トークンで中央寄せ）。サイドバーの「ホーム」はホーム画面のURL（`/`）へリンクする。表示内容のログイン前/後は、ホーム側で認証状態を参照して判定する。

### 2.2 成果物

- 共通で使うレイアウトコンポーネント（例: `HomeLayout`, `TrialLayout`）の枠だけでもよいので用意する。
- 左カラム幅をトークン（`--content-left-max-width: 948px`）で定義し、ホーム・トライアルの両方で参照する（いちおしサイト用に content-right 328px のため 78px 縮小）。

## 【3】ホーム画面の実装

**目的**: プロトタイプ SCREEN-000（ホームページ v1.1）の構成を Next 上で再現する。

### 3.1 やること

- バナー、「最新動画」「本日の一番」「昨日までの積重ね」（ログイン後のみ）、「最新記事」、右カラム（参考リンク・広告エリア）など、プロトタイプのブロックをコンポーネントに分割して実装する。
- 表示切替: **今の段階ではモック**でよい。URL パラメータ（例: `?logged_in=true`）やローカル state で「ログイン前／後風」を切り替え、本日の一番・昨日までの積重ね等の表示／非表示を再現する。DB・認証実装時には、ここを AuthContext 等の認証状態に差し替える。
- スタイル: content-left の最大幅 948px、main-content 最大 1300px、右カラム 328px を守る。

### 3.2 成果物

- ホーム用ページコンポーネント（`app/page.tsx`。`/` がホームページ）。
- 必要なら「こころのトライアル」へのリンクを、トライアルの入口URL（`/trial_4w`）に向ける。

## 【4】トライアル4画面の実装

**目的**: アファメーション・朝・晩・週・月の4画面を、プロトタイプの構成と仕様に合わせて実装する。

### 4.1 やること

- **1ページ**（`/trial_4w`）で、メニューバーの選択に応じて「アファメーション」「朝・晩」「週」「月」の4コンテンツを**クライアント側で切り替える**（案2）。
- **共通仕様**
  - trial-main の最大幅は **948px**（`--content-left-max-width`。ホーム content-left と同一）。
  - サイドバー「ホーム」はホーム画面のURL（`/`）へリンクする。ログイン表示はホーム側で認証状態を参照して判定する。
- メニューバー（アファメーション / 朝・晩 / 週 / 月）のクリックで、表示するコンポーネントを state で切替。必要なら URL クエリ（`?tab=affirmation` 等）と同期し、タブの復元・共有ができるようにする。
- プロトタイプの入力項目・ラベル・レイアウトをできる範囲で再現する（まずは静的な構成でよい）。

### 4.2 成果物

- **1ページ**: `app/trial_4w/page.tsx`（共通レイアウト＋メインエリアのタブ切替）。
- **4つのメインコンテンツコンポーネント**: アファメーション、朝・晩、週、月（各プロトタイプ相当）。これらを state またはクエリに応じて出し分ける。
- 共通の TrialLayout（ヘッダー・サイドバー・メニューバー。trial-main 幅 948px）。`layout.tsx` にしても、同一 `page.tsx` 内に含めてもよい。

## 【5】ローカルで動作確認

**目的**: 実装したホーム・トライアルが、仕様どおり動くか手元で確認する。

### 5.1 やること

- `npm run dev` で開発サーバーを起動する。
- 次を確認する:
  - ホーム: 未ログイン時・ログイン後（または `?logged_in=true`）で表示が切り替わるか。
  - ホームの左カラム幅が 948px で収まっているか（ブラウザの開発者ツールで幅を確認）。
  - トライアル各画面: trial-main が 948px で中央寄せになっているか。
  - トライアルの「ホーム」リンクが、ホーム画面のURL（`/`）になっているか。表示のログイン前/後は、ホーム側の認証状態で切り替わるか。
- 可能ならスマートフォン幅でもレイアウトが崩れないか確認する。

### 5.2 成果物

- 不具合や仕様ずれのメモ。必要なら【3】【4】に戻って修正する。

## 【6】ビルドして外で確認

**目的**: 本番に近い形でビルドし、外部（別端末・別ネットワーク・デプロイ先）で表示と動作を確認する。

### 6.1 やること

- `npm run build` でビルドが通るか確認する。エラーがあれば解消する。
- **外で見る方法の例**:
  - **A. 同じマシンで本番モード**: `npm run start` で起動し、ブラウザで `http://localhost:3000` を開く。必要なら同一LAN内の別端末から `http://<PCのIP>:3000` でアクセスする。
  - **B. デプロイ**: Vercel / Firebase Hosting / その他にデプロイし、本番URLで確認する。
  - **C. 静的エクスポート**: Next を静的エクスポートする設定にしている場合は、生成された静的ファイルを任意の Web サーバーで配信し、そのURLで確認する。

### 6.2 確認ポイント

- ホーム・トライアルの表示がローカルと同様に意図どおりか。
- 「ホーム」でホーム画面に遷移し、認証済みならログイン後の表示になっているか。
- 実機（スマホ・タブレット）で見たときのレイアウトやタップ操作。

### 6.3 成果物

- 外で確認した結果のメモ。問題があれば【3】〜【5】に戻って修正し、必要に応じて再度ビルド・確認する。
