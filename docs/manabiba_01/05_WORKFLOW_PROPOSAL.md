# 設計書・実装の進め方 - 作業フロー提案

## 📋 目的

要求仕様の変更に伴い、設計書と実装を段階的に進めるための作業フローを検討します。

---

## 1. 提案されたフロー

### 1.1 基本フロー

```
1. ユーザー → 要件定義書を修正
2. Cursor → その他の設計書を修正（整合性確保）
3. ユーザー → 検証
4. 実装前の内部設計書作成
5. 内部仕様の検証
6. 実装修正
```

### 1.2 各ステップの詳細

#### ステップ1: 要件定義書の修正
- **担当**: ユーザー
- **作業内容**: 
  - 新しい要件の追加
  - 既存要件の変更
  - 要件の削除
- **成果物**: 更新された要件定義書

#### ステップ2: その他の設計書の修正
- **担当**: Cursor（AI）
- **作業内容**:
  - システムアーキテクチャ設計書の更新
  - 機能一覧の更新
  - ユーザーフローの更新
  - データモデルの更新
- **成果物**: 整合性が取れた設計書一式

#### ステップ3: 検証
- **担当**: ユーザー
- **作業内容**:
  - 設計書の整合性確認
  - 要件の漏れがないか確認
  - 技術的な実現可能性の確認
- **成果物**: 承認または修正指示

#### ステップ4: 内部設計書の作成
- **担当**: Cursor（AI）または ユーザー
- **作業内容**:
  - コンポーネント設計書の作成
  - API設計書の作成
  - データベース設計書の詳細化
- **成果物**: 内部設計書

#### ステップ5: 内部仕様の検証
- **担当**: ユーザー
- **作業内容**:
  - 内部設計の妥当性確認
  - 実装可能性の確認
- **成果物**: 承認または修正指示

#### ステップ6: 実装修正
- **担当**: ユーザー + Cursor（AI）
- **作業内容**:
  - コードの実装
  - テスト
  - デバッグ
- **成果物**: 実装された機能

---

## 2. フローの評価

### 2.1 提案フローのメリット

✅ **段階的な進捗**
- 一つずつ確実に進められる
- 混乱を避けられる

✅ **責任の明確化**
- 誰が何をするか明確
- 検証ポイントが明確

✅ **整合性の確保**
- 設計書間の整合性を自動的に保つ
- 矛盾を早期に発見

✅ **実装前の検証**
- 内部設計書で実装前に検証
- 実装後の手戻りを減らす

### 2.2 提案フローのデメリット・課題

⚠️ **時間がかかる**
- 段階的に進めるため、全体の進捗が遅くなる可能性

⚠️ **内部設計書の粒度**
- どの程度詳細に書くか判断が必要
- 過度に詳細だと時間がかかる

⚠️ **変更の連鎖**
- 要件変更が複数の設計書に影響する場合、修正が多くなる

## 3. 推奨フロー（改良版）

### 3.1 基本フロー

```
┌─────────────────────────────────────────┐
│ フェーズ1: 要件定義・外部設計            │
├─────────────────────────────────────────┤
│ 1. ユーザー → 要件定義書を修正          │
│ 2. Cursor → 外部設計書を修正            │
│    - システムアーキテクチャ設計書        │
│    - 機能一覧                            │
│    - ユーザーフロー                      │
│ 3. ユーザー → 検証・承認                │
└─────────────────────────────────────────┘
           ↓
┌─────────────────────────────────────────┐
│ フェーズ2: 内部設計                      │
├─────────────────────────────────────────┤
│ 4. Cursor → 内部設計書を作成            │
│    - コンポーネント設計書                │
│    - API設計書                           │
│    - データベース設計書（詳細）          │
│ 5. ユーザー → 内部設計の検証             │
└─────────────────────────────────────────┘
           ↓
┌─────────────────────────────────────────┐
│ フェーズ3: 実装                          │
├─────────────────────────────────────────┤
│ 6. 実装修正（ユーザー + Cursor）        │
│ 7. テスト                                │
│ 8. 設計書の更新（実装内容を反映）        │
└─────────────────────────────────────────┘
```

### 3.2 詳細フロー

#### フェーズ1: 要件定義・外部設計（1-2週間）

**ステップ1: 要件定義書の修正**
- ユーザーが要件定義書を修正
- 変更内容を明確に記録（変更履歴に追加）

**ステップ2: 外部設計書の自動修正**
- Cursorが要件定義書の変更を検出
- 影響を受ける設計書を自動的に修正
  - システムアーキテクチャ設計書
  - 機能一覧
  - ユーザーフロー
  - データモデル（必要に応じて）

**ステップ3: 検証・承認**
- ユーザーが設計書の整合性を確認
- 必要に応じて修正指示

**チェックポイント**:
- [ ] 要件定義書の変更がすべて反映されている
- [ ] 外部設計書の整合性が保たれている
- [ ] 技術的な実現可能性が確認されている

#### フェーズ2: 内部設計（1週間）

**ステップ4: 内部設計書の作成**
- 実装する機能に必要な内部設計書を作成
  - コンポーネント設計書（新規・変更コンポーネント）
  - API設計書（Firestore操作の詳細）
  - データベース設計書（詳細なスキーマ）

**内部設計書の粒度**:
- **必須**: コンポーネントの責務、Props、State
- **必須**: Firestore操作の詳細（関数名、パラメータ、戻り値）
- **推奨**: 処理フロー（シーケンス図）
- **任意**: 詳細な実装ロジック（必要に応じて）

**ステップ5: 内部設計の検証**
- ユーザーが内部設計を確認
- 実装可能性を検証
- パフォーマンス、セキュリティの確認

**チェックポイント**:
- [ ] コンポーネント設計が明確
- [ ] API設計が明確
- [ ] データベース設計が明確
- [ ] 実装可能であることが確認されている

#### フェーズ3: 実装（2-4週間）

**ステップ6: 実装修正**
- 内部設計書に基づいて実装
- 段階的に実装（機能ごと）

**ステップ7: テスト**
- 単体テスト
- 統合テスト
- ユーザーテスト

**ステップ8: 設計書の更新**
- 実装内容を設計書に反映
- 実装と設計書の整合性を保つ

## 4. 内部設計書の必要性

### 4.1 内部設計書が必要な場合

✅ **新規機能の実装**
- 新しいコンポーネントを作成する場合
- 新しいFirestore操作を追加する場合

✅ **既存機能の大幅な変更**
- 既存コンポーネントの責務が変わる場合
- データモデルが変わる場合

✅ **複雑な機能**
- 複数のコンポーネントが関わる場合
- 複雑な状態管理が必要な場合

### 4.2 内部設計書が不要な場合

❌ **軽微な修正**
- バグ修正
- UIの微調整
- 文言の変更

❌ **既存パターンの繰り返し**
- 既存のコンポーネントと同じパターンで実装する場合

### 4.3 推奨アプローチ

**段階的な内部設計**:
1. **最初**: コンポーネント設計書のみ作成（Props、State、責務）
2. **必要に応じて**: 詳細な処理フローを追加
3. **複雑な場合**: シーケンス図を追加

**テンプレート化**:
- 内部設計書のテンプレートを作成
- 必要最小限の情報を記入
- 過度に詳細にしない

## 5. 整合性管理の方法

### 5.1 設計書間の依存関係

```
要件定義書
    ↓
システムアーキテクチャ設計書
    ↓
機能一覧
    ↓
ユーザーフロー
    ↓
内部設計書（コンポーネント、API、DB）
    ↓
実装
```

### 5.2 整合性チェックリスト

**要件定義書変更時**:
- [ ] システムアーキテクチャ設計書を更新
- [ ] 機能一覧を更新
- [ ] ユーザーフローを更新
- [ ] データモデルを更新（必要に応じて）

**内部設計書作成時**:
- [ ] 要件定義書の要件を満たしている
- [ ] システムアーキテクチャに準拠している
- [ ] 既存のコンポーネントとの整合性

**実装時**:
- [ ] 内部設計書通りに実装されている
- [ ] 要件定義書の要件を満たしている
- [ ] 設計書を実装内容に合わせて更新

### 5.3 自動化の可能性

**Cursor（AI）の役割**:
- 要件定義書の変更を検出
- 影響を受ける設計書を自動修正
- 整合性チェックリストを自動生成

**ユーザーの役割**:
- 要件定義書の修正
- 設計書の検証・承認
- 実装の検証

## 6. 実践的なワークフロー

### 6.1 小さな変更の場合

```
1. ユーザー → 要件定義書を修正
2. Cursor → 影響を受ける設計書を自動修正
3. ユーザー → 検証
4. 実装（内部設計書は簡易版のみ）
```

### 6.2 大きな変更の場合

```
1. ユーザー → 要件定義書を修正
2. Cursor → 外部設計書を修正
3. ユーザー → 検証
4. Cursor → 内部設計書を作成（詳細版）
5. ユーザー → 内部設計の検証
6. 実装
7. 設計書の更新
```

### 6.3 新規機能追加の場合

```
1. ユーザー → 要件定義書に新機能を追加
2. Cursor → 外部設計書を更新
3. ユーザー → 検証
4. Cursor → 内部設計書を作成（新規機能）
5. ユーザー → 内部設計の検証
6. 実装
7. 設計書の更新
```

## 7. 推奨事項

### 7.1 設計書の粒度

**外部設計書**:
- 機能の概要、入力・出力、画面遷移
- 詳細な実装ロジックは含めない

**内部設計書**:
- コンポーネントの責務、Props、State
- Firestore操作の詳細
- 必要に応じて処理フロー

### 7.2 変更管理

**変更履歴の記録**:
- 各設計書に変更履歴を記録
- 変更理由を明確に

**バージョン管理**:
- Gitで管理
- 変更内容をコミットメッセージに記録

### 7.3 検証のタイミング

**外部設計の検証**:
- 要件定義書変更後、すぐに検証
- 整合性を確認

**内部設計の検証**:
- 実装前に必ず検証
- 実装可能性を確認

**実装後の検証**:
- 実装内容が設計書通りか確認
- 設計書を実装内容に合わせて更新

## 8. 質問への回答

### Q1: 全体の整合性を都度修正した方が良いですか？

**回答**: **はい、推奨します**

**理由**:
- 設計書間の矛盾を早期に発見できる
- 実装時の混乱を避けられる
- 変更の影響範囲を明確にできる

**方法**:
- 要件定義書変更後、Cursorが自動的に他の設計書を修正
- ユーザーが検証して承認

### Q2: 実装前に内部設計書を作成すべきですか？

**回答**: **機能の複雑さに応じて判断**

**必須の場合**:
- 新規機能の実装
- 既存機能の大幅な変更
- 複雑な機能

**不要の場合**:
- 軽微な修正
- 既存パターンの繰り返し

**推奨アプローチ**:
- 最初は簡易版（コンポーネント設計のみ）
- 必要に応じて詳細化

### Q3: 実装前に内部仕様の検証は必要ですか？

**回答**: **はい、推奨します**

**理由**:
- 実装可能性を確認できる
- 実装後の手戻りを減らせる
- パフォーマンス、セキュリティの問題を早期に発見

**検証内容**:
- 技術的な実現可能性
- パフォーマンスへの影響
- セキュリティへの影響
- 既存機能への影響
